# 三层 C/S 架构介绍

## 一句话理解

三层 C/S 架构是在传统客户端/服务器架构基础上，把系统划分为：

```text
表示层 Client
业务逻辑层 Application Server
数据层 Database Server
```

客户端不直接访问数据库，而是通过中间的应用服务器完成业务处理，再由应用服务器访问数据库。

## 三层结构

```text
客户端/表示层
  |
  | 请求、响应、界面交互
  v
应用服务器/业务逻辑层
  |
  | 业务规则、权限校验、事务控制、接口封装
  v
数据库服务器/数据层
     数据存储、查询、更新、备份
```

## 各层职责

### 1. 表示层

表示层也叫客户端层，主要负责用户交互和界面展示。

常见职责：

- 展示界面。
- 接收用户输入。
- 做简单格式校验。
- 向业务逻辑层发送请求。
- 展示业务逻辑层返回的结果。

在你的工作场景中，Unity 或 Cocos 开发的手机客户端就属于表示层。

### 2. 业务逻辑层

业务逻辑层通常部署在应用服务器上，是三层 C/S 架构的核心。

常见职责：

- 处理业务规则。
- 进行身份认证和权限校验。
- 控制事务。
- 调用数据层读写数据。
- 封装接口，避免客户端直接操作数据库。
- 处理日志、缓存、限流、异常等公共逻辑。

例如游戏服务端中的账号服务、角色服务、商城服务、活动服务、排行榜服务等，都可以看作业务逻辑层的一部分。

### 3. 数据层

数据层主要负责数据持久化和数据访问。

常见职责：

- 存储业务数据。
- 执行查询、插入、更新、删除。
- 维护索引和约束。
- 进行备份、恢复和容灾。
- 保证数据完整性和一致性。

例如 MySQL 数据库就属于数据层。

## 与两层 C/S 架构的区别

两层 C/S 架构通常是：

```text
客户端 -> 数据库服务器
```

客户端直接连接数据库，部分甚至大量业务逻辑写在客户端中。

三层 C/S 架构则是：

```text
客户端 -> 应用服务器 -> 数据库服务器
```

业务逻辑集中在应用服务器中，客户端不直接访问数据库。

## 三层 C/S 架构的优点

### 1. 降低客户端复杂度

客户端主要负责界面展示和交互，不需要承担大量业务逻辑，也不需要直接连接数据库。

这样客户端更轻量，升级和维护更方便。

### 2. 提高安全性

客户端不能直接访问数据库，数据库账号、表结构、核心数据不会暴露给客户端。

所有数据访问都经过业务逻辑层，可以统一做身份认证、权限控制、接口签名和安全审计。

### 3. 提高可维护性

业务规则集中在应用服务器上。如果业务规则变化，只需要修改服务端逻辑，不必频繁更新所有客户端。

例如游戏活动规则、商城折扣、道具发放逻辑发生变化时，可以主要在服务端调整。

### 4. 提高可扩展性

业务逻辑层可以横向扩展。

当访问量增大时，可以增加应用服务器实例，通过负载均衡分担请求压力。

### 5. 便于事务和一致性控制

涉及多个数据表或多个业务步骤时，业务逻辑层可以统一控制事务。

例如购买道具时，需要同时扣除货币、增加道具、记录订单，这些步骤适合在服务端统一处理。

## 三层 C/S 架构的缺点

### 1. 系统部署更复杂

相比两层 C/S 架构，三层 C/S 架构多了应用服务器，部署、监控、运维都更复杂。

### 2. 可能增加通信开销

请求需要经过客户端、应用服务器、数据库服务器，多了一层网络调用。

如果设计不当，可能增加响应时间。

### 3. 应用服务器可能成为瓶颈

业务逻辑层承担了大量请求处理任务。如果没有做负载均衡、缓存、限流和扩容，应用服务器可能成为性能瓶颈。

## 手游业务样例

以一个 Unity/Cocos 手机游戏的“玩家购买道具”为例。

### 业务场景

玩家在游戏客户端中点击购买一个道具，例如“高级宝箱”。

系统需要完成：

- 检查玩家是否登录。
- 校验商品是否存在。
- 判断玩家货币是否足够。
- 扣除货币。
- 发放道具。
- 写入订单记录。
- 返回购买结果。

### 三层 C/S 处理流程

```text
1. Unity/Cocos 客户端
   玩家点击购买按钮
   客户端向商城服务发送购买请求

2. 应用服务器
   校验玩家登录状态
   校验请求签名和参数
   查询商品配置
   判断玩家余额是否足够
   执行扣款和发放道具逻辑
   生成订单流水

3. MySQL 数据库
   更新玩家货币表
   更新玩家背包表
   插入订单记录

4. 应用服务器
   组装购买结果
   记录业务日志
   返回响应

5. Unity/Cocos 客户端
   展示购买成功或失败提示
   刷新玩家货币和背包界面
```

### 对应三层职责

```text
表示层：
  Unity/Cocos 客户端，负责展示商城界面和发送购买请求。

业务逻辑层：
  商城服务，负责登录校验、商品校验、余额判断、扣款、发道具、订单生成。

数据层：
  MySQL，负责保存玩家货币、背包、订单等数据。
```

## 考试答题模板

如果考试问“三层 C/S 架构的组成和特点”，可以这样答：

```text
三层 C/S 架构将系统划分为表示层、业务逻辑层和数据层。
表示层部署在客户端，负责用户界面、输入输出和结果展示；
业务逻辑层部署在应用服务器，负责业务规则处理、权限控制、事务管理和数据访问封装；
数据层部署在数据库服务器，负责数据存储、查询、更新和维护。

与两层 C/S 相比，三层 C/S 架构使客户端不直接访问数据库，
降低了客户端复杂度，提高了系统安全性、可维护性和可扩展性。
但它也增加了系统部署和通信开销，应用服务器可能成为性能瓶颈。
```
