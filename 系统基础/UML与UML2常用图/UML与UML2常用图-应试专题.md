```text

```

容易混淆时，先问一句：

```text
题目问“谁和系统交互”      -> 用例图
```

## 2. UML 是什么

UML 是统一建模语言。

它是：

```text
一种建模语言
一种可视化表达方式
一种用于描述、构造、文档化系统的标准符号体系
```

它不是：

```text
不是程序设计语言
不是开发方法学
不是某一种具体开发过程
```

软考常考判断：

```text
UML 与程序设计语言无关。
UML 支持面向对象分析与设计。
UML 也可用于需求分析到设计实现的全过程。
UML 是语言，不是方法。
```

## 3. UML 的组成

教材中把 UML 的组成理解为：

```text
基本构造块
图
公共机制
```

基本构造块又包括：

```text
事物
关系
```

## 4. UML 中的事物

UML 事物分为四类：

```text
结构事物
行为事物
分组事物
注释事物
```

### 结构事物

结构事物是模型中的“名词”，描述静态元素。

常见包括：

```text
类
接口
协作
用例
主动类
构件
制品
结点
```

记忆：

```text
结构事物 = 静态名词
```

### 行为事物

行为事物是模型中的“动词”，描述动态行为。

常见包括：

```text
交互
状态机
活动
```

记忆：

```text
行为事物 = 动态过程
```

### 分组事物

最典型的是：

```text
包 Package
```

作用：

```text
把模型元素组织成组
```

### 注释事物

最典型的是：

```text
注解 Note
```

作用：

```text
解释、约束、说明模型元素
```

## 5. UML 中的四种关系

UML 基本关系：

```text
依赖
关联
泛化
实现
```

### 依赖 Dependency

含义：

```text
A 的变化可能影响 B
```

图形：

```text
虚线箭头
```

常见关键词：

```text
使用
依赖
变化影响
```

### 关联 Association

含义：

```text
对象之间有结构连接
```

图形：

```text
实线
```

常带：

```text
角色名
多重度
```

多重度例子：

```text
1
0..1
0..*
1..*
*
```

### 聚合 Aggregation

聚合是关联的特殊形式。

含义：

```text
整体-部分关系，部分可以脱离整体存在
```

图形：

```text
空心菱形
```

例子：

```text
班级 - 学生
```

### 组合 Composition

组合是更强的整体-部分关系。

含义：

```text
部分依赖整体生命周期
整体消失，部分通常也消失
```

图形：

```text
实心菱形
```

例子：

```text
订单 - 订单明细
```

### 泛化 Generalization

含义：

```text
继承关系
子类是一种父类
```

图形：

```text
实线 + 空心三角箭头
箭头指向父类
```

### 实现 Realization

含义：

```text
类实现接口
构件实现接口
```

图形：

```text
虚线 + 空心三角箭头
箭头指向接口
```

## 7. 用例图

用例图描述：

```text
系统从外部看能提供什么功能
谁使用这些功能
系统边界在哪里
```

核心元素：

```text
参与者 Actor
用例 Use Case
系统边界
参与者与用例之间的关联
用例之间的包含、扩展、泛化
```

### 用例图用途

用于：

```text
需求分析
系统语境建模
系统功能需求建模
黑盒视角描述系统
```

不用来描述：

```text
类的属性和方法
对象之间消息顺序
程序部署在哪台机器
内部算法流程细节
```

### 用例图三种高频关系

用例与用例之间常考：

```text
包含 include
扩展 extend
泛化 generalization
```

注意：

```text
聚合不是用例图关系
聚合属于类图中的整体-部分关系
```

### include 包含关系

含义：

```text
公共的、必做的、复用的子用例
```

判断词：

```text
总是执行
必须执行
公共步骤
多个用例复用
```

例子：

```text
下单 -> include -> 用户登录
支付 -> include -> 安全校验
```

### extend 扩展关系

含义：

```text
可选的、条件触发的、额外行为
```

判断词：

```text
可选
条件满足时
异常流程
附加功能
```

例子：

```text
下单 <- extend <- 使用优惠券
登录 <- extend <- 短信验证码
```

### 泛化关系

含义：

```text
特殊用例继承一般用例
特殊参与者继承一般参与者
```

例子：

```text
管理员 是 用户 的一种
微信支付 是 支付 的一种
```

## 8. 活动图

活动图描述：

```text
业务流程
工作流
控制流
动作之间的先后、分支、并发
```

它关注：

```text
事情怎么一步步流转
```

常见元素：

```text
初始节点
活动/动作
控制流
判断节点
合并节点
分叉 fork
汇合 join
终止节点
泳道
```

活动图适合描述：

```text
订单处理流程
审批流程
登录认证流程
业务流程建模
算法或工作流步骤
```

活动图不重点描述：

```text
对象之间的消息时间顺序
类之间的继承和关联
对象生命周期状态变化
```

与流程图的关系：

```text
活动图可以理解为 UML 中更规范的流程建模图
支持分支、并发、泳道
```

## 9. 顺序图/序列图

顺序图也常叫：

```text
序列图
时序图
Sequence Diagram
```

它描述：

```text
对象之间消息传递的时间顺序
```

核心元素：

```text
对象/参与者
生命线 Lifeline
消息 Message
激活 Activation
返回消息
```

读图方式：

```text
从上往下读时间
横向看对象
竖向看生命线
箭头看消息调用
```

适合描述：

```text
一个用例的具体执行场景
对象之间如何协作
接口调用先后顺序
运行时交互过程
```

顺序图不重点描述：

```text
业务流程分支的整体结构
类的静态关系
部署节点
对象状态机
```

## 10. 顺序图 vs 活动图

最容易混：

```text
活动图：流程怎么走
顺序图：对象之间怎么发消息
```

例子：玩家购买道具。

活动图关心：

```text
选择道具
检查余额
扣款
发放道具
记录日志
```

顺序图关心：

```text
客户端 -> 游戏服：购买请求
游戏服 -> 支付服务：扣款
游戏服 -> 背包服务：发放道具
游戏服 -> 日志服务：写日志
游戏服 -> 客户端：返回结果
```

考试判断：

```text
出现“消息、生命线、交互、时间顺序” -> 顺序图
出现“流程、活动、分支、并发、泳道” -> 活动图
```

## 11. 类图

类图描述：

```text
系统的静态结构
类、接口、属性、操作
类与类之间的关系
```

类图常用于：

```text
领域模型
设计模型
静态结构建模
数据库实体关系的面向对象表达
```

类的表示通常包括：

```text
类名
属性
方法/操作
```

可见性：

```text
+ public
- private
# protected
```

类图高频关系：

```text
关联
聚合
组合
泛化
实现
依赖
```

易错点：

```text
聚合、组合主要在类图中考
不要把聚合当成用例图关系
```

## 12. 状态图

状态图描述：

```text
一个对象在生命周期中的状态变化
```

核心元素：

```text
状态
事件
转换
初态
终态
```

适合描述：

```text
订单状态变化
角色在线/离线状态
任务状态
设备状态
```

例子：

```text
订单：
待支付 -> 已支付 -> 已发货 -> 已完成
待支付 -> 已取消
```

和活动图区分：

```text
状态图：对象处于什么状态，以及状态怎么变
活动图：流程中执行哪些活动，以及活动怎么流转
```

## 13. 构件图

构件图描述：

```text
软件构件、模块、接口及它们之间的依赖关系
```

适合描述：

```text
实现视图
代码构件组织
模块依赖
组件接口
```

关键词：

```text
构件
组件
接口
模块
实现
依赖
```

软考判断：

```text
问软件实现模块怎么组织 -> 构件图
```

## 14. 部署图

部署图描述：

```text
软件制品部署到哪些硬件节点
节点之间如何连接
系统的物理拓扑
```

适合描述：

```text
服务器
客户端
数据库服务器
容器节点
网络连接
物理部署架构
```

关键词：

```text
节点
设备
执行环境
制品
物理部署
网络连接
```

软考判断：

```text
问程序跑在哪台机器上 -> 部署图
```

## 15. 其他 UML 图速记

对象图：

```text
类图的实例快照
某一时刻对象及对象间链接
```

通信图：

```text
对象之间交互
强调对象连接关系
消息通常带编号
```

包图：

```text
模型元素分组
包与包之间依赖
```

组合结构图：

```text
描述类或构件的内部结构
```

交互概览图：

```text
活动图 + 交互图的结合
用于概览多个交互流程
```

计时图：

```text
关注对象状态或值随时间变化
实时系统题目中容易出现
```

## 17. UML 与 MBSE、SysML

教材在 MBSE 中也提到 UML 2.0。

关系可以这样记：

```text
MBSE = 基于模型的系统工程
SysML = 系统建模语言
SysML 基于 UML 2.0 的子集进行重用和扩展
```

MBSE 三大支柱：

```text
建模语言
建模工具
建模思路
```

MBSE 阶段与图：

```text
需求分析：
需求图、用例图、包图

功能分析与分配：
顺序图、活动图、状态机图

设计综合：
模块定义图、内部块图、参数图
```

软考判断：

```text
看到 SysML，想到系统工程建模，不是普通 UML 的简单改名。
看到 UML 2.0 子集扩展，想到 SysML。
```

## 18. 易混点总表

```text
用例图 vs 活动图
用例图看系统提供什么功能；活动图看流程怎么走。

活动图 vs 顺序图
活动图看活动流；顺序图看对象消息时间顺序。

顺序图 vs 通信图
顺序图强调时间顺序；通信图强调对象连接和消息编号。

活动图 vs 状态图
活动图看流程；状态图看对象生命周期状态变化。

构件图 vs 部署图
构件图看软件模块；部署图看物理节点和部署位置。

聚合/组合 vs 包含/扩展
聚合/组合属于类图；包含/扩展属于用例图。
```

## 19. 常见题型 1：UML 性质判断

题目：

```text
关于 UML 的描述，正确的是（ ）。

A UML 是一种程序设计语言
B UML 是一种开发方法学
C UML 是一种统一建模语言
D UML 只能用于面向对象设计阶段
```

答案：

```text
C
```

解析：

```text
UML 是建模语言，不是程序设计语言，也不是开发方法学。
它可支持需求、分析、设计等多个阶段。
```

## 23. 常见题型 5：include 与 extend

题目：

```text

B extend
C aggregation
D realization
```

答案：


解析：

```text
每次都必须执行、公共复用步骤 -> include。
```

## 31. 高频预测题 1

## 41. 最终背诵版

```text
UML 是建模语言，不是程序语言，也不是方法学。

UML 事物：
结构、行为、分组、注释。

UML 关系：
依赖、关联、泛化、实现。

UML 2.0 交互图：
顺序图、通信图、交互概览图、计时图。

用例图：
参与者、用例、系统边界；
用例关系是 include、extend、泛化；
聚合不是用例图关系。

活动图：
业务流程、控制流、分支、并发、泳道。

顺序图：
对象消息、生命线、时间顺序。

类图：
类、属性、方法、关联、聚合、组合、泛化、实现。

状态图：
对象生命周期状态变化。

构件图：
软件模块和接口依赖。

部署图：
硬件节点和物理部署。
```
