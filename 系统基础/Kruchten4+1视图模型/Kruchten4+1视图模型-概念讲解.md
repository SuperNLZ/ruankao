# Kruchten 4+1 视图模型-概念讲解

## 1. 概述

Kruchten 提出的“4+1”视图模型是一种软件架构描述方法，用 5 个不同视角描述软件系统架构。

其中“4”指四个基本架构视图：

```text
逻辑视图
开发视图
进程视图
物理视图
```

“+1”指：

```text
场景视图
```

场景视图通常用用例或典型场景来串联和验证前四个视图。

## 2. 逻辑视图

逻辑视图关注系统向用户提供的功能。

它主要描述：

- 系统的核心功能。
- 主要抽象、类、对象和它们之间的关系。
- 业务逻辑结构。

面向对象方法中，逻辑视图常用类图、对象图等表示。

适合回答：

```text
系统提供什么功能？
核心业务对象有哪些？
对象之间有什么关系？
```

## 3. 开发视图

开发视图也称实现视图。

它关注软件在开发环境中的组织方式。

它主要描述：

- 模块划分。
- 子系统划分。
- 包、组件、源代码结构。
- 分层结构和依赖关系。

开发视图主要面向开发人员，用于指导代码组织、模块边界和团队分工。

适合回答：

```text
代码如何组织？
系统如何分模块？
模块之间如何依赖？
```

## 4. 进程视图

进程视图关注系统运行时的并发、通信和性能。

它主要描述：

- 进程和线程。
- 并发控制。
- 进程间通信。
- 同步机制。
- 性能、吞吐量、可伸缩性。

进程视图主要用于分析系统运行时行为。

适合回答：

```text
系统运行时有哪些进程或线程？
它们如何通信和同步？
系统如何满足性能和并发要求？
```

## 5. 物理视图

物理视图也称部署视图。

它关注软件到硬件环境的映射关系。

它主要描述：

- 服务器、终端、网络节点。
- 软件组件部署在哪些物理节点上。
- 节点之间的网络连接。
- 部署结构、容灾和扩展方式。

物理视图主要面向系统工程师和运维人员。

适合回答：

```text
系统部署在哪些机器上？
组件如何映射到硬件节点？
节点之间如何通信？
```

## 6. 场景视图

场景视图是“4+1”中的“+1”。

它通过用例或典型业务场景描述系统如何完成某个具体任务。

场景视图的作用是：

- 串联逻辑视图、开发视图、进程视图和物理视图。
- 验证架构设计是否满足需求。
- 帮助理解系统在真实场景下的协作过程。

适合回答：

```text
一个典型业务流程如何贯穿整个系统？
架构设计能否支撑关键用例？
```

## 7. 五个视图的关系

四个基本视图分别从功能、开发、运行和部署角度描述系统。

场景视图则通过具体用例把这些视图联系起来。

可以理解为：

```text
逻辑视图：看功能
开发视图：看代码
进程视图：看运行
物理视图：看部署
场景视图：看用例验证
```

## 8. 举例

以在线游戏服务系统为例：

- 逻辑视图：账号、角色、背包、商城、排行榜等业务对象。
- 开发视图：账号服务、角色服务、商城服务、排行榜服务、公共组件库。
- 进程视图：网关进程、业务服务进程、数据库连接池、异步任务线程。
- 物理视图：客户端、网关服务器、业务服务器、数据库服务器、缓存服务器。
- 场景视图：玩家登录、购买道具、参与活动、刷新排行榜等典型用例。

## 9. 考试答题模板

如果题目要求说明“4+1”视图模型，可按下面结构回答：

```text
Kruchten 的 4+1 视图模型从多个角度描述软件架构。
其中 4 个基本视图包括逻辑视图、开发视图、进程视图和物理视图，
分别关注系统功能、软件模块组织、运行时并发通信以及物理部署。
+1 是场景视图，通常通过用例或典型场景来验证和串联前四个视图，
从而保证架构设计能够满足关键需求。
```

## 10. 常考点

1. “4+1”视图模型由 Kruchten 提出。
2. 四个基本视图是逻辑视图、开发视图、进程视图、物理视图。
3. “+1”是场景视图，也常称用例视图。
4. 逻辑视图关注功能和对象模型。
5. 开发视图关注模块、组件和代码组织。
6. 进程视图关注运行时进程、线程、通信和性能。
7. 物理视图关注部署、节点和网络拓扑。
8. 场景视图用于验证和串联其他视图。
