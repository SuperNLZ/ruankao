# 断路器模式-快速记忆

## 一句话

```text
断路器模式 = 调用外部服务连续失败时，先“断开调用”，避免故障扩散。
```

它常用于：

```text
微服务、分布式系统、远程接口调用、第三方服务调用
```

## 核心目的

```text
防止故障级联
保护系统资源
提升系统可用性
```

## 三种状态

```text
Closed    关闭：正常调用
Open      打开：直接失败，不再调用远程服务
Half-Open 半开：放少量请求试探服务是否恢复
```

## 状态口诀

```text
关着能通，开了拒绝，半开试探
```

对应关系：

```text
Closed    -> 正常放行
Open      -> 快速失败
Half-Open -> 少量试探
```

## 状态转换

```text
正常调用
  -> 失败次数过多
  -> Open 打开断路器
  -> 等待一段时间
  -> Half-Open 半开试探
  -> 成功则恢复 Closed
  -> 失败则重新 Open
```

## 看到这些词就想到断路器

| 关键词 | 判断 |
|---|---|
| 服务连续失败 | 断路器 |
| 快速失败 | 断路器 |
| 防止级联故障 | 断路器 |
| 熔断 | 断路器 |
| 半开试探 | 断路器 |
| 保护下游服务 | 断路器 |
| 远程调用失败过多 | 断路器 |

## 和其他模式区分

### 断路器 vs 重试

```text
重试：失败后再试几次。
断路器：失败太多后先别试，直接快速失败。
```

### 断路器 vs 限流

```text
限流：请求太多，限制进入。
断路器：下游不行，停止调用。
```

### 断路器 vs 降级

```text
降级：返回简化结果或备用结果。
断路器：阻止继续调用故障服务。
```

## 游戏服务例子

```text
游戏服调用支付服务。
如果支付服务连续超时，断路器打开。
后续购买请求不再继续访问支付服务，而是快速返回“支付服务繁忙”。
等待一段时间后，进入半开状态，放少量请求试探。
如果支付服务恢复成功，断路器关闭，恢复正常调用。
```

## 易错点

```text
断路器不是为了提高单次调用成功率，而是为了避免故障扩散。
断路器打开后不是继续重试，而是快速失败。
半开状态不是完全恢复，而是少量请求试探。
断路器通常和重试、限流、降级一起使用。
```

## 最终背诵版

```text
断路器模式用于分布式调用中，当外部服务连续失败或超时时，
暂时切断调用并快速失败，防止故障级联和资源耗尽。
它有 Closed、Open、Half-Open 三种状态：
Closed 正常调用，Open 快速失败，Half-Open 少量试探恢复。
```
