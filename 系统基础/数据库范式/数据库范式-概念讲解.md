# 数据库范式-概念讲解

## 1. 范式解决什么问题

数据库范式是关系数据库设计中的规范化规则，目标是让表结构更合理，减少数据冗余，并避免插入异常、删除异常和修改异常。

简单说：

```text
范式 = 判断一张表设计得是否合理的一组规则。
规范化 = 按范式要求拆表。
```

如果一张表把多个主题混在一起，例如学生信息、课程信息、选课成绩都放在同一张表中，就容易出现重复数据和维护问题。范式化的基本方法就是把不同主题拆到不同表中，再通过主键、外键关联。

## 2. 三类异常

### 插入异常

有些数据暂时无法插入。

例子：

```text
选课表(学号, 学生名, 课程号, 课程名, 成绩)
```

如果一个新课程还没有学生选课，就没有学号，课程信息可能无法单独插入。

### 删除异常

删除一条记录时，把本不该删除的信息也删掉。

例子：

```text
如果某门课程只有一个学生选课，删除该学生选课记录后，课程名也丢失。
```

### 修改异常

同一信息重复存储，修改时必须改多处。

例子：

```text
课程名在多条选课记录中重复出现。
课程改名时，每条记录都要修改，漏改就会不一致。
```

## 3. 先理解函数依赖

函数依赖是判断范式的基础。

```text
X -> Y
```

表示：只要 X 的值确定，Y 的值也就确定。

例子：

```text
学号 -> 学生名
课程号 -> 课程名
(学号, 课程号) -> 成绩
```

含义：

```text
知道学号，就能确定学生名。
知道课程号，就能确定课程名。
知道学号和课程号，才能确定这名学生这门课的成绩。
```

## 4. 码相关概念

### 超码

能唯一标识一行的属性集合。

```text
学生(学号, 身份证号, 姓名)
```

学号可以唯一标识学生，身份证号也可以唯一标识学生。

```text
学号
身份证号
(学号, 姓名)
```

这些都可能是超码。

### 候选码

候选码是最小的超码。也就是说，它能唯一标识一行，并且不能再去掉任何属性。

```text
学号 是候选码
身份证号 是候选码
(学号, 姓名) 不是候选码，因为学号单独就够了
```

### 主码

从候选码中选一个作为主码。

```text
候选码可以有多个，主码通常选一个。
```

### 主属性和非主属性

```text
主属性：出现在任一候选码中的属性。
非主属性：不出现在任何候选码中的属性。
```

考试中判断 2NF 和 3NF 时，经常会用到“非主属性”。

## 5. 第一范式 1NF

1NF 要求关系中的每个属性都是不可再分的原子值。

违反 1NF 的典型情况：

```text
学生(学号, 姓名, 电话)
电话 = "13800000000, 13900000000"
```

一个字段中放了多个电话号码，这个字段就不是原子值。

改法：

```text
学生(学号, 姓名)
学生电话(学号, 电话)
```

考试关键词：

```text
属性不可再分
字段原子性
一个单元格不能存多个值
```

## 6. 第二范式 2NF

2NF 的前提是已经满足 1NF。

2NF 要求：每个非主属性都必须完全依赖候选码，不能只依赖候选码的一部分。

它主要针对组合候选码。

例子：

```text
选课(学号, 课程号, 学生名, 课程名, 成绩)
候选码 = (学号, 课程号)
```

函数依赖：

```text
学号 -> 学生名
课程号 -> 课程名
(学号, 课程号) -> 成绩
```

分析：

```text
学生名 只依赖 学号，而不是完整的 (学号, 课程号)
课程名 只依赖 课程号，而不是完整的 (学号, 课程号)
成绩 依赖完整的 (学号, 课程号)
```

所以：

```text
学生名、课程名存在部分依赖
该表违反 2NF
```

拆分：

```text
学生(学号, 学生名)
课程(课程号, 课程名)
选课(学号, 课程号, 成绩)
```

考试关键词：

```text
组合码
部分函数依赖
非主属性只依赖主码的一部分
```

## 7. 第三范式 3NF

3NF 的前提是已经满足 2NF。

3NF 要求：非主属性不能传递依赖候选码。

典型形式：

```text
候选码 -> 非主属性A
非主属性A -> 非主属性B
所以 候选码 -> 非主属性B 是传递依赖
```

例子：

```text
学生(学号, 姓名, 系号, 系名)
候选码 = 学号
```

函数依赖：

```text
学号 -> 姓名
学号 -> 系号
系号 -> 系名
```

分析：

```text
学号 -> 系号
系号 -> 系名
因此 学号 -> 系名 是传递依赖
```

所以该表违反 3NF。

拆分：

```text
学生(学号, 姓名, 系号)
院系(系号, 系名)
```

考试关键词：

```text
传递函数依赖
非主属性决定非主属性
候选码 -> A -> B
```

## 8. BCNF

BCNF 比 3NF 更严格。

BCNF 要求：对于关系中的每一个非平凡函数依赖 X -> Y，X 都必须是候选码或超码。

简单判断：

```text
只要某个决定因素 X 不是候选码/超码，就违反 BCNF。
```

这里的“决定因素”就是函数依赖左边的属性集合。

3NF 和 BCNF 的区别：

```text
3NF 重点看非主属性是否传递依赖候选码。
BCNF 重点看每个函数依赖左边是不是码。
```

考试中如果出现：

```text
X -> Y
X 不是候选码
```

就优先想到：

```text
违反 BCNF
```

## 9. 第四范式 4NF

4NF 主要处理多值依赖问题。它比 BCNF 更进一步，不只看函数依赖，还看一张表里是否混入了多个互相独立的一对多事实。

先记一个形式：

```text
X ->> Y
```

它表示多值依赖：对于同一个 X，Y 可以有多个值。

例子：

```text
学生爱好电话(学号, 爱好, 电话)
```

如果业务规则是：

```text
一个学生可以有多个爱好
一个学生可以有多个电话
爱好和电话之间没有关系
```

那么对同一个学号来说：

```text
学号 ->> 爱好
学号 ->> 电话
```

这就是多值依赖。

如果张三有两个爱好、两个电话，这张表会被迫保存所有组合：

```text
张三, 篮球, 13800000000
张三, 篮球, 13900000000
张三, 游泳, 13800000000
张三, 游泳, 13900000000
```

这里的问题不是某一格存了多个值。每一格仍然是原子值，所以它可以满足 1NF。真正的问题是：同一张表里同时混合了“学生-爱好”和“学生-电话”两组彼此独立的一对多关系，导致组合冗余。

4NF 的要求：

```text
对于每一个非平凡多值依赖 X ->> Y，X 必须是候选码或超码。
```

简单判断：

```text
存在独立多值事实
并且决定因素不是候选码/超码
就违反 4NF
```

拆分方式：

```text
学生爱好(学号, 爱好)
学生电话(学号, 电话)
```

这样就不用把爱好和电话做笛卡尔组合。

4NF 和 1NF 的区别：

```text
1NF：一个单元格里不能放多个值。
4NF：一张表里不能混放多个独立的多值关系。
```

考试关键词：

```text
多值依赖
非平凡多值依赖
互相独立的一对多关系
消除组合冗余
```

## 10. 范式之间的关系

范式是逐级增强的：

```text
4NF -> BCNF -> 3NF -> 2NF -> 1NF
```

也就是说：

```text
满足 4NF 一定满足 BCNF
满足 BCNF 一定满足 3NF
满足 3NF 一定满足 2NF
满足 2NF 一定满足 1NF
```

反过来不一定成立：

```text
满足 1NF 不一定满足 2NF
满足 2NF 不一定满足 3NF
满足 3NF 不一定满足 BCNF
满足 BCNF 不一定满足 4NF
```

## 11. 规范化拆分思路

看到一张大表时，可以按下面顺序判断：

```text
1. 字段是否原子？
   不原子 -> 违反 1NF

2. 候选码是否是组合码？
   是组合码 -> 检查非主属性是否只依赖其中一部分
   有部分依赖 -> 违反 2NF

3. 是否存在 非主属性 -> 非主属性？
   存在候选码 -> A -> B
   有传递依赖 -> 违反 3NF

4. 每个函数依赖左边是不是候选码/超码？
   不是 -> 违反 BCNF

5. 是否存在多个互相独立的一对多事实？
   有非平凡多值依赖 -> 违反 4NF
```

拆分原则：

```text
依赖谁，就把谁和被它决定的属性拆成一张表。
```

例如：

```text
学号 -> 学生名      拆成 学生(学号, 学生名)
课程号 -> 课程名    拆成 课程(课程号, 课程名)
系号 -> 系名        拆成 院系(系号, 系名)
学号 ->> 爱好       拆成 学生爱好(学号, 爱好)
学号 ->> 电话       拆成 学生电话(学号, 电话)
```

## 12. 软考常见问法

常见题干：

```text
关系模式 R(...)
函数依赖集 F = {...}
问该关系最高满足第几范式
问违反了哪个范式
问应该如何分解
问是否存在部分依赖/传递依赖
问是否存在多值依赖
```

快速处理步骤：

```text
1. 先求候选码。
2. 区分主属性和非主属性。
3. 看是否有部分依赖，判断 2NF。
4. 看是否有传递依赖，判断 3NF。
5. 看每个决定因素是否为候选码/超码，判断 BCNF。
6. 看是否存在独立多值事实，判断 4NF。
```

## 13. 总结

```text
1NF：属性必须原子。
2NF：在 1NF 基础上，消除非主属性对组合候选码的部分依赖。
3NF：在 2NF 基础上，消除非主属性对候选码的传递依赖。
BCNF：每个函数依赖的决定因素都必须是候选码或超码。
4NF：消除非平凡多值依赖，避免多个独立多值事实混在一张表中。
```

最终考试记忆：

```text
一原子，二完全，三传递，BCNF看决定，四多值。
```
