# 白盒测试介绍

## 基本概念

白盒测试又称结构测试、逻辑驱动测试或基于程序结构的测试。

它把被测程序看成一个透明的盒子，测试人员需要了解程序内部结构、控制流程、数据流和代码逻辑，并根据这些内部信息设计测试用例。

白盒测试关注的是：

```text
程序内部逻辑是否正确
程序路径是否被充分执行
条件判断是否覆盖完整
循环和分支是否存在缺陷
代码中是否存在不可达路径或逻辑错误
```

## 与黑盒测试的区别

| 对比项 | 白盒测试 | 黑盒测试 |
|---|---|---|
| 测试依据 | 程序内部结构、代码逻辑 | 需求规格说明、输入输出关系 |
| 是否了解代码 | 需要了解 | 不需要了解 |
| 关注重点 | 路径、分支、条件、语句、数据流 | 功能、边界、等价类、业务规则 |
| 常用方法 | 语句覆盖、判定覆盖、条件覆盖、路径覆盖 | 等价类划分、边界值分析、因果图、判定表 |
| 适用阶段 | 单元测试、集成测试较常见 | 系统测试、验收测试较常见 |

简记：

```text
白盒看内部逻辑
黑盒看外部功能
```

## 白盒测试的主要目的

白盒测试主要用于发现程序内部逻辑问题，包括：

- 语句执行错误。
- 判断条件错误。
- 分支逻辑遗漏。
- 循环边界错误。
- 路径组合错误。
- 数据流异常。
- 异常处理缺失。
- 死代码或不可达代码。

## 常见覆盖标准

白盒测试中最常考的是覆盖标准。覆盖强度通常从弱到强逐步提高。

### 1. 语句覆盖

语句覆盖要求：

```text
程序中的每条可执行语句至少执行一次
```

优点是简单，缺点是覆盖能力较弱。

例如：

```text
if (a > 0 && b > 0) {
    x = 1;
}
```

只要设计一个 `a > 0` 且 `b > 0` 的测试用例，就可以让 `x = 1` 执行，从而满足语句覆盖。

但是它不一定能发现条件组合错误。

### 2. 判定覆盖

判定覆盖也称分支覆盖。

它要求：

```text
每个判定的真分支和假分支都至少执行一次
```

例如：

```text
if (a > 0) {
    x = 1;
} else {
    x = 2;
}
```

需要至少设计两类用例：

```text
a > 0   走真分支
a <= 0  走假分支
```

判定覆盖比语句覆盖更强。

### 3. 条件覆盖

条件覆盖要求：

```text
每个判定中的每个条件，其真值和假值都至少出现一次
```

例如：

```text
if (a > 0 && b > 0)
```

其中包含两个条件：

```text
a > 0
b > 0
```

条件覆盖要求：

```text
a > 0 为真、为假都出现
b > 0 为真、为假都出现
```

注意：条件覆盖不一定保证每个判定结果的真假都出现。

### 4. 判定/条件覆盖

判定/条件覆盖要求同时满足：

```text
每个判定结果的真、假都至少出现一次
每个条件结果的真、假都至少出现一次
```

它比单独的判定覆盖和条件覆盖更强。

### 5. 条件组合覆盖

条件组合覆盖要求：

```text
每个判定中所有条件取值组合都至少出现一次
```

例如：

```text
if (A && B)
```

两个条件 `A` 和 `B` 的组合有：

```text
A=true,  B=true
A=true,  B=false
A=false, B=true
A=false, B=false
```

因此至少要覆盖这 4 种组合。

条件组合覆盖比判定/条件覆盖更强，但测试用例数量可能明显增加。

### 6. 路径覆盖

路径覆盖要求：

```text
程序中所有可能的执行路径都至少执行一次
```

路径覆盖是更强的覆盖标准，但在存在循环和复杂分支时，路径数量可能非常多，甚至不可穷尽。

实际测试中通常会选择重要路径、基本路径或独立路径进行覆盖。

## 覆盖强度大致关系

一般可以这样理解：

```text
语句覆盖 < 判定覆盖 < 判定/条件覆盖 < 条件组合覆盖
路径覆盖通常更强，但复杂程序中难以完全实现
```

需要注意：

```text
条件覆盖不一定强于判定覆盖
判定覆盖也不一定强于条件覆盖
```

因为二者关注点不同：

- 判定覆盖关注整个判定结果的真和假。
- 条件覆盖关注判定内部每个条件的真和假。

## 基本路径测试

基本路径测试是白盒测试中的重要方法。

它基于程序控制流图，计算程序的环路复杂度，然后设计测试用例覆盖所有独立路径。

### 控制流图

控制流图用节点和边表示程序控制结构：

- 节点表示程序语句或语句块。
- 边表示控制流转移。

### 环路复杂度

环路复杂度用于度量程序逻辑复杂度，也表示独立路径的数量。

常用公式：

```text
V(G) = E - N + 2
```

其中：

```text
E 表示控制流图中的边数
N 表示控制流图中的节点数
```

也可用：

```text
V(G) = 判定节点数 + 1
```

在单入口、单出口的结构化程序中，后一种方法更适合快速做题。

### 基本路径测试步骤

```text
1. 根据程序画出控制流图
2. 计算环路复杂度 V(G)
3. 找出 V(G) 条独立路径
4. 为每条独立路径设计测试用例
```

## 常见白盒测试方法

### 逻辑覆盖法

逻辑覆盖法包括：

- 语句覆盖。
- 判定覆盖。
- 条件覆盖。
- 判定/条件覆盖。
- 条件组合覆盖。
- 路径覆盖。

这是软考中最常见的考点。

### 基本路径测试法

基本路径测试法常与控制流图、环路复杂度一起考。

考试中常见问法：

```text
根据程序流程图计算环路复杂度
至少需要设计多少条独立路径
至少需要多少个测试用例
```

### 数据流测试

数据流测试关注变量的定义和使用情况。

常见检查点包括：

- 变量定义后未使用。
- 变量使用前未定义。
- 变量重复定义。
- 变量定义后路径异常。

## 白盒测试适用场景

白盒测试常用于：

- 单元测试。
- 模块测试。
- 部分集成测试。
- 核心算法测试。
- 安全敏感代码测试。
- 高可靠性代码测试。

在实际开发中，开发人员通常更适合执行白盒测试，因为他们了解代码结构和内部逻辑。

## 白盒测试的优缺点

### 优点

- 可以深入检查程序内部逻辑。
- 能发现隐藏在代码中的路径错误。
- 适合发现边界、分支和循环问题。
- 有助于提高代码覆盖率。
- 可用于评估测试充分性。

### 缺点

- 需要了解代码，实现成本较高。
- 对测试人员技术要求较高。
- 难以发现需求遗漏类问题。
- 路径数量多时，完全覆盖不现实。
- 不能替代黑盒测试。

## 软考高频考点

软考中白盒测试常考以下内容：

1. 白盒测试与黑盒测试的区别。
2. 哪些方法属于白盒测试。
3. 各种覆盖标准的含义。
4. 不同覆盖标准之间的强弱关系。
5. 控制流图和环路复杂度计算。
6. 根据覆盖要求选择最少测试用例。
7. 基本路径测试的步骤。

## 易错点

1. 语句覆盖最弱，不能保证分支都被执行。
2. 判定覆盖要求每个判定结果真假都出现。
3. 条件覆盖要求每个条件真假都出现，但不一定覆盖判定真假。
4. 条件组合覆盖要求所有条件组合都出现。
5. 路径覆盖很强，但复杂程序中通常难以完全实现。
6. 白盒测试不是只测功能，而是根据程序内部结构设计测试。
7. 白盒测试不能完全替代黑盒测试，二者关注点不同。

## 记忆口诀

```text
白盒看代码，黑盒看功能；
语句看执行，判定看真假；
条件看子项，组合看全配；
路径看流程，复杂度看独立路径数。
```
