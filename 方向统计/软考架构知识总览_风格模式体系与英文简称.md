# 软考架构知识总览（风格 / 模式 / 体系 / 英文简称）

# 一、架构风格（Architecture Style）

## 1. 调用/返回风格（Call-and-Return）

核心：

```text
上层调用下层
```

典型：

- 主程序-子程序
- RPC
- 分层架构

特点：

```text
控制流驱动
```

用途：

```text
适合层次清晰、调用关系明确的传统业务系统
```

---

## 2. 管道-过滤器风格（Pipe-Filter）

核心：

```text
数据流水线加工
```

组成：

- Pipe（传输）
- Filter（处理）

典型：

- Linux |
- 编译器
- ETL

用途：

```text
适合连续数据处理、流水线加工、批量转换场景
```

---

## 3. 黑板风格（Blackboard）

核心：

```text
多个专家共享黑板协同求解
```

典型：

- AI系统
- 语音识别
- 专家系统

特点：

```text
协同推理
```

用途：

```text
适合复杂推理、不确定求解、AI专家系统
```

---

## 4. 仓库风格（Repository）

核心：

```text
多个模块共享中央数据仓库
```

组成：

- Repository
- Component
- Connector

典型：

- 数据库系统
- ERP
- 编译器符号表

用途：

```text
适合多个模块共享统一数据的数据中心型系统
```

---

## 5. 进程通信风格（Communicating Process）

核心：

```text
多个独立进程通过消息通信
```

典型：

- 微服务
- 分布式系统
- Actor模型

特点：

```text
并发 + 通信
```

用途：

```text
适合并发处理、分布式系统、进程间协作
```

---

## 6. 微服务架构（Microservices）

核心：

```text
系统拆分为多个独立小服务
```

特点：

- 独立部署
- 独立扩展
- 服务通信

典型：

- 电商系统
- 云平台

用途：

```text
适合大型复杂系统、快速迭代、独立部署与弹性扩展
```

---

## 7. Lambda 架构（Lambda Architecture）

核心：

```text
批处理 + 流处理
```

组成：

- Batch Layer
- Speed Layer
- Serving Layer

典型：

- 大数据实时分析

用途：

```text
适合同时需要离线全量计算和实时增量计算的大数据系统
```

---

# 二、架构模式（Architecture Pattern）

## 1. MVC（Model-View-Controller）

核心：

```text
数据、界面、控制逻辑分离
```

组成：

- Model
- View
- Controller

特点：

```text
View 可直接访问 Model
```

用途：

```text
适合传统 GUI、Web 应用的界面与业务分离
```

---

## 2. MVP（Model-View-Presenter）

核心：

```text
Presenter 负责业务逻辑
```

组成：

- Model
- View
- Presenter

特点：

```text
View 不直接操作 Model
```

用途：

```text
适合需要降低 UI 与业务逻辑耦合、方便测试的界面系统
```

---

## 3. 三层 C/S 架构

核心：

```text
表示层 + 业务层 + 数据层
```

三层：

- UI层
- 业务逻辑层
- 数据访问层

特点：

```text
业务逻辑独立
```

用途：

```text
适合企业管理系统、客户端/服务器系统的分层部署
```

---

# 三、架构评估与演化

## 1. ATAM

全称：

```text
Architecture Tradeoff Analysis Method
```

核心：

```text
分析质量属性之间的权衡
ATAM 在设计之初，主要关注的质量属性包括性能、实用性（或可用性）、安全性和可修改性
```

重点：

- 效用树
- 风险点
- 权衡点

用途：

```text
适合评估架构是否满足性能、安全、可用性等质量属性
```

---

## 2. SAAM

全称：

```text
Software Architecture Analysis Method
```

核心：

```text
可修改性分析
```

用途：

```text
适合分析架构面对需求变化时的修改影响
```

---

## 3. CBAM

全称：

```text
Cost Benefit Analysis Method
```

核心：

```text
成本收益分析
```

用途：

```text
适合从经济角度评估架构方案的投入与收益
```

# 4.DSSA（领域特定软件体系结构）

全称：

```text
Domain Specific Software Architecture
```

核心：

```text
针对某一领域建立可复用架构与构件
```

三阶段：

| 阶段     | 核心目标       |
| -------- | -------------- |
| 领域分析 | 获得领域模型   |
| 领域设计 | 获得 DSSA      |
| 领域实现 | 开发可复用构件 |

用途：

```
适合银行、电商、航空等具有大量共性需求的领域系统
```

---

## 4. 效用树（Utility Tree）

核心：

```text
组织质量属性场景
```

结构：

```text
树根
↓
质量属性
↓
属性分类
↓
质量属性场景
```

用途：

```text
适合把抽象质量需求拆成可评估、可排序的具体场景
```

---

## 5. 体系结构演化

核心：

```text
随着需求变化不断修改架构
```

六步骤：

```text
归类
计划
变构件
改交互
组装测试
技术评审
```

用途：

```text
适合在需求变化后有计划地修改和验证软件体系结构
```

# 六、最容易混淆的内容（超高频）

# 1. 黑板 vs 仓库

| 对比 | 黑板 | 仓库 |
|---|---|---|
| 核心 | 协同推理 | 数据共享 |
| 重点 | 共同解题 | 中央数据 |
| 典型 | AI系统 | 数据库系统 |

口诀：

```text
黑板解题
仓库存数据
```

---

# 2. 管道过滤器 vs 调用返回

| 对比 | 管道过滤器 | 调用返回 |
|---|---|---|
| 核心 | 数据流 | 控制流 |
| 通信 | Pipe | 函数调用 |
| 典型 | ETL | 分层系统 |

口诀：

```text
管道传数据
调用传控制
```

---

# 3. MVC vs MVP

| 对比 | MVC | MVP |
|---|---|---|
| View 是否直接访问 Model | 可以 | 不可以 |
| 业务逻辑 | Controller | Presenter |
| 耦合度 | 较高 | 较低 |

口诀：

```text
MVC View 可碰 Model
MVP 必须走 Presenter
```

---

# 4. SAAM vs ATAM vs CBAM

| 方法 | 核心 |
|---|---|
| SAAM | 可修改性分析 |
| ATAM | 质量属性权衡 |
| CBAM | 成本收益分析 |

口诀：

```text
SAAM 看修改
ATAM 看权衡
CBAM 看成本
```

# ABSD 方法（极简速记）

# 一、真题

基于架构的软件设计（ABSD）方法由（ ）驱动体系结构设计，并递归细化直到产生（ ）。

答案：

```text
第一空：B 商业、质量和功能需求
第二空：B 软件构件和类
```

------

# 二、方法核心（高频）

- 自顶向下、递归细化架构
- 基于系统功能分解
- 选择架构风格满足质量和商业需求
- 使用软件模板

------

# 三、开发基础（软考高频）

1. 功能分解（模块内聚与耦合）
2. 架构风格选择（质量和商业需求）
3. 软件模板使用

------

# 四、软考口诀

```text
ABSD：
商业+质量+功能驱动
递归到构件和类
选风格用模板
```

------

# 五、一句话总结

> ABSD 方法以商业、质量和功能需求驱动架构设计，自顶向下递归细化，最终产生软件构件和类，并通过架构风格与软件模板实现系统目标。





# 磁盘调度：移臂调度与旋转调度（极简速记）

# 一、题目关键句

```text
先进行移臂调度
再进行旋转调度
```

含义：

```text
先确定访问哪个柱面
再确定同柱面内访问哪个扇区
```

------

# 二、移臂调度看什么

```text
看柱面号
```

最短移臂调度算法：

```text
每次选择距离当前磁头所在柱面最近的柱面
```

例如当前在 20 号柱面：

```text
20 → 21 → 22 → 18 → 16
```

------

# 三、旋转调度看什么

```text
看扇区号
```

当多个请求在同一个柱面上时：

```text
按扇区号从小到大读
```

前提：

```text
题目没有给当前扇区位置和旋转方向
```

软考默认：

```text
同柱面内扇区号小的先读
```

------

# 四、例子

18 号柱面上有：

| 请求 | 扇区号 |
| ---- | ------ |
| ⑤    | 4      |
| ⑦    | 6      |
| ①    | 6      |

按扇区号从小到大：

```text
⑤ → ⑦ → ①
```

------

# 五、完整解题步骤

```text
1. 先按柱面号做移臂调度
2. 得到柱面访问顺序
3. 再在同一柱面内按扇区号排序
4. 合并得到最终响应序列
```

------

# 六、口诀

```text
移臂看柱面
旋转看扇区
同柱面内小扇区先读
```

------

# 七、高频易错点

| 易错点                 | 正确理解                                 |
| ---------------------- | ---------------------------------------- |
| 只按请求编号排序       | 错，先看柱面                             |
| 只按 SSTF 一个个请求选 | 错，同柱面还要旋转调度                   |
| 同柱面按磁头号排序     | 错，旋转调度看扇区号                     |
| 真实磁盘一定从小到大   | 不一定；软考题未给当前扇区时默认从小到大 |

------

# 八、一句话总结

> 磁盘调度题若说明“先移臂调度，再旋转调度”，则先按柱面号确定访问顺序，再在同一柱面内按扇区号从小到大确定请求顺序。
