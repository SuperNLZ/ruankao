# 软考架构知识总览（风格 / 模式 / 体系 / 英文简称）

# 一、架构风格（Architecture Style）

## 1. 调用/返回风格（Call-and-Return）

核心：

```text
上层调用下层
```

典型：

- 主程序-子程序
- RPC
- 分层架构

特点：

```text
控制流驱动
```

用途：

```text
适合层次清晰、调用关系明确的传统业务系统
```

---

## 2. 管道-过滤器风格（Pipe-Filter）

核心：

```text
数据流水线加工
```

组成：

- Pipe（传输）
- Filter（处理）

典型：

- Linux |
- 编译器
- ETL

用途：

```text
适合连续数据处理、流水线加工、批量转换场景
```

---

## 3. 黑板风格（Blackboard）

核心：

```text
多个专家共享黑板协同求解
```

典型：

- AI系统
- 语音识别
- 专家系统

特点：

```text
协同推理
```

用途：

```text
适合复杂推理、不确定求解、AI专家系统
```

---

## 4. 仓库风格（Repository）

核心：

```text
多个模块共享中央数据仓库
```

组成：

- Repository
- Component
- Connector

典型：

- 数据库系统
- ERP
- 编译器符号表

用途：

```text
适合多个模块共享统一数据的数据中心型系统
```

---

## 5. 进程通信风格（Communicating Process）

核心：

```text
多个独立进程通过消息通信
```

典型：

- 微服务
- 分布式系统
- Actor模型

特点：

```text
并发 + 通信
```

用途：

```text
适合并发处理、分布式系统、进程间协作
```

---

## 6. 微服务架构（Microservices）

核心：

```text
系统拆分为多个独立小服务
```

特点：

- 独立部署
- 独立扩展
- 服务通信

典型：

- 电商系统
- 云平台

用途：

```text
适合大型复杂系统、快速迭代、独立部署与弹性扩展
```

---

## 7. Lambda 架构（Lambda Architecture）

核心：

```text
批处理 + 流处理
```

组成：

- Batch Layer
- Speed Layer
- Serving Layer

典型：

- 大数据实时分析

用途：

```text
适合同时需要离线全量计算和实时增量计算的大数据系统
```

---

# 二、架构模式（Architecture Pattern）

## 1. MVC（Model-View-Controller）

核心：

```text
数据、界面、控制逻辑分离
```

组成：

- Model
- View
- Controller

特点：

```text
View 可直接访问 Model
```

用途：

```text
适合传统 GUI、Web 应用的界面与业务分离
```

---

## 2. MVP（Model-View-Presenter）

核心：

```text
Presenter 负责业务逻辑
```

组成：

- Model
- View
- Presenter

特点：

```text
View 不直接操作 Model
```

用途：

```text
适合需要降低 UI 与业务逻辑耦合、方便测试的界面系统
```

---

## 3. 三层 C/S 架构

核心：

```text
表示层 + 业务层 + 数据层
```

三层：

- UI层
- 业务逻辑层
- 数据访问层

特点：

```text
业务逻辑独立
```

用途：

```text
适合企业管理系统、客户端/服务器系统的分层部署
```

---

# 三、架构评估与演化

## 1. ATAM

全称：

```text
Architecture Tradeoff Analysis Method
```

核心：

```text
分析质量属性之间的权衡
```

重点：

- 效用树
- 风险点
- 权衡点

用途：

```text
适合评估架构是否满足性能、安全、可用性等质量属性
```

---

## 2. SAAM

全称：

```text
Software Architecture Analysis Method
```

核心：

```text
可修改性分析
```

用途：

```text
适合分析架构面对需求变化时的修改影响
```

---

## 3. CBAM

全称：

```text
Cost Benefit Analysis Method
```

核心：

```text
成本收益分析
```

用途：

```text
适合从经济角度评估架构方案的投入与收益
```

---

## 4. 效用树（Utility Tree）

核心：

```text
组织质量属性场景
```

结构：

```text
树根
↓
质量属性
↓
属性分类
↓
质量属性场景
```

用途：

```text
适合把抽象质量需求拆成可评估、可排序的具体场景
```

---

## 5. 体系结构演化

核心：

```text
随着需求变化不断修改架构
```

六步骤：

```text
归类
计划
变构件
改交互
组装测试
技术评审
```

用途：

```text
适合在需求变化后有计划地修改和验证软件体系结构
```

# 五、最容易混淆的内容（超高频）

# 1. 黑板 vs 仓库

| 对比 | 黑板 | 仓库 |
|---|---|---|
| 核心 | 协同推理 | 数据共享 |
| 重点 | 共同解题 | 中央数据 |
| 典型 | AI系统 | 数据库系统 |

口诀：

```text
黑板解题
仓库存数据
```

---

# 2. 管道过滤器 vs 调用返回

| 对比 | 管道过滤器 | 调用返回 |
|---|---|---|
| 核心 | 数据流 | 控制流 |
| 通信 | Pipe | 函数调用 |
| 典型 | ETL | 分层系统 |

口诀：

```text
管道传数据
调用传控制
```

---

# 3. MVC vs MVP

| 对比 | MVC | MVP |
|---|---|---|
| View 是否直接访问 Model | 可以 | 不可以 |
| 业务逻辑 | Controller | Presenter |
| 耦合度 | 较高 | 较低 |

口诀：

```text
MVC View 可碰 Model
MVP 必须走 Presenter
```

---

# 4. SAAM vs ATAM vs CBAM

| 方法 | 核心 |
|---|---|
| SAAM | 可修改性分析 |
| ATAM | 质量属性权衡 |
| CBAM | 成本收益分析 |

口诀：

```text
SAAM 看修改
ATAM 看权衡
CBAM 看成本
```

---
