## 3. 主题优先级和背诵关键词

### 3.1 第一优先级：大模型 / RAG / 知识图谱智能问答架构

押题理由：

```text
新大纲和近年案例题明显偏向 AI、大模型、RAG、知识图谱、向量数据库。
论文尚未充分覆盖该方向，适合以新技术题面出现。
```

必须背的关键词：

```text
大模型
生成式 AI
RAG
检索增强生成
文档解析
知识切分
数据清洗
元数据标注
向量化
向量数据库
语义检索
关键词检索
混合检索
知识图谱
实体关系
检索结果重排
Prompt 编排
模型网关
引用来源
幻觉治理
权限过滤
敏感信息脱敏
审计日志
离线评测集
人工反馈闭环
```

考场关键句：

```text
大模型应用不能只依赖模型能力，必须通过知识治理、检索增强、权限控制和持续评测形成可控工程体系。
```

三段式写法：

```text
1. 先做企业知识治理：采集、清洗、切分、标注、向量化。
2. 再做 RAG 问答流程：认证、问题改写、混合检索、重排、Prompt 拼接、模型生成、引用返回。
3. 最后做风险治理：防幻觉、防泄露、审计、评测、反馈闭环。
```

### 3.2 第二优先级：系统安全架构设计

押题理由：

```text
安全是架构师核心质量属性，可与账号、支付、运营后台、AI 数据安全、零信任结合。
近年安全单独成题概率仍值得重视。
```

必须背的关键词：

```text
身份认证
Token
会话管理
API 网关
统一鉴权
RBAC
最小权限
接口签名
时间戳
随机数
防重放
参数校验
支付回调签名校验
订单幂等
金额校验
敏感数据脱敏
加密存储
传输加密
日志审计
安全告警
限流防刷
异常检测
零信任
```

考场关键句：

```text
安全架构不是某个模块的附加功能，而应贯穿接入层、业务层、数据层和运维层全过程。
```

三段式写法：

```text
1. 接入安全：认证、Token、网关鉴权、签名、防重放、限流。
2. 业务安全：支付回调校验、订单幂等、道具发放补偿、后台 RBAC。
3. 数据与运维安全：脱敏、加密、审计、告警、权限复核。
```

### 3.3 第三优先级：高可用 / 可靠性 / 容错架构

押题理由：

```text
高可用是系统架构设计师最稳的母题之一，能覆盖可靠性、容错、故障恢复、负载均衡、韧性设计。
移动游戏项目非常容易套。
```

必须背的关键词：

```text
服务集群
负载均衡
健康检查
故障摘除
多实例部署
故障隔离
限流
熔断
降级
重试
超时控制
幂等
消息队列削峰
异步处理
数据库主从
备份恢复
监控告警
链路追踪
MTTR
接口成功率
错误率
补偿机制
```

考场关键句：

```text
高可用架构不是简单增加服务器，而是围绕核心链路建立冗余、隔离、削峰、恢复和可观测能力。
```

三段式写法：

```text
1. 接入层高可用：负载均衡、健康检查、限流、故障摘除。
2. 服务层高可用：集群部署、熔断降级、幂等、消息队列、故障隔离。
3. 数据和运维高可用：数据库主从、备份恢复、监控告警、补偿和演练。
```

### 3.4 第四优先级：微服务 / 服务治理 / 云原生架构

押题理由：

```text
微服务、服务治理、云原生是近年常考架构方向。
虽然部分相关主题已经考过，但换成服务治理、可观测性、云原生改造仍可能出现。
```

必须背的关键词：

```text
服务拆分
业务边界
领域建模
API 网关
服务注册与发现
配置中心
服务治理
接口契约
链路追踪
日志
指标
可观测性
容器化
Kubernetes
弹性伸缩
CI/CD
蓝绿发布
灰度发布
金丝雀发布
限流熔断降级
事件驱动
消息队列
最终一致性
分布式事务
```

考场关键句：

```text
微服务架构的关键不是把系统拆得越细越好，而是按业务边界拆分，并配套服务治理、自动化交付和可观测能力。
```

三段式写法：

```text
1. 服务拆分：按账号、角色、商城、活动、排行榜、邮件、日志等业务边界拆分。
2. 服务治理：注册发现、网关、配置、限流熔断、链路追踪、监控告警。
3. 工程保障：容器化、CI/CD、灰度发布、事件驱动、最终一致性。
```

## 4. 考场选题判断

看到题目后先匹配关键词。

| 题目关键词 | 优先套用 |
|---|---|
| 人工智能、大模型、智能问答、知识库、RAG、向量数据库、知识图谱 | 01 大模型 RAG 知识图谱 |
| 安全、认证、授权、数据保护、零信任、接口安全、后台权限 | 02 系统安全架构 |
| 高可用、可靠性、容错、故障恢复、韧性、负载均衡 | 03 高可用可靠性容错 |
| 微服务、服务治理、云原生、服务拆分、可观测性、服务网格 | 04 微服务服务治理云原生 |

如果四题都能写，优先选择：

```text
大模型/RAG > 安全架构 > 高可用 > 微服务/云原生
```

原因：

```text
大模型方向新，区分度高；
安全和高可用是质量属性母题，容易写实；
微服务/云原生要避免写成泛泛技术堆砌。
```

## 5. 最后 30 分钟背诵清单

每篇只背 4 件事：

```text
1. 摘要第一段。
2. 项目背景第一段。
3. 三个关键措施标题。
4. 总结句。
```

统一摘要开头可复用：

```text
我参与设计并建设了一套面向移动游戏的高并发在线服务平台，该平台服务于多款 Unity/Cocos 开发的手机游戏，主要包括账号登录、角色管理、商城支付、活动配置、排行榜、邮件通知、数据统计和运营后台等模块。
```

统一收尾句可复用：

```text
实践表明，架构设计不能只追求某个单点技术，而应结合业务特点，在性能、可用性、安全性、可扩展性、可维护性和实施成本之间进行合理权衡。
```
