# 论高可用架构_修改版

## 题目

论高可用架构设计

## 考场提纲

一、项目背景

- 面向移动游戏的高并发在线服务平台。
- 服务多款 Unity/Cocos 手机游戏。
- 核心业务包括登录、角色、组队匹配、排行榜、商城支付、邮件发放、数据统计和运营后台。
- 开服、版本更新和限时活动时并发高，支付和发奖链路长，对高可用要求高。

二、高可用目标

- 核心业务持续可用。
- 局部故障隔离，避免故障扩散。
- 高峰流量可承受，避免系统雪崩。
- 关键数据可追踪、可补偿、可恢复。
- 故障可发现、可定位、可处理。

三、关键设计措施

- 接入层采用负载均衡、健康检查和故障摘除。
- 服务层采用 Orleans Actor、服务拆分和多实例部署。
- 流量层采用限流、消息队列和削峰填谷。
- 依赖层采用超时、重试、熔断和降级。
- 数据层采用 MySQL 主从、备份、幂等和补偿。
- 运维层采用监控告警、日志追踪、灰度发布和故障演练。

四、实施过程中的问题

- 消息重复和顺序问题。
- 支付和道具发放链路复杂。
- 服务拆分后链路变长，故障定位困难。
- 缓存、数据库和异步任务存在一致性问题。

五、实施效果和总结

- 活动高峰稳定性提升。
- 单实例故障影响范围缩小。
- 支付和发奖链路可追踪、可补偿。
- 故障发现和恢复时间缩短。
- 高可用设计需要在可用性、成本、复杂度和一致性之间权衡。

## 摘要

我参与设计并建设了一套面向移动游戏的高并发在线服务平台，该平台服务于多款 Unity/Cocos 开发的手机游戏，主要包括玩家登录、角色管理、组队匹配、排行榜、商城支付、邮件发放、数据统计和运营后台等模块。随着游戏用户规模扩大，系统在开服、版本更新、限时活动和支付高峰时经常面临瞬时流量大、服务实例故障、数据库压力集中、下游服务异常扩散以及关键数据恢复要求高等问题。为了保证核心业务稳定运行，我们围绕高可用目标，从接入层、服务层、流量层、数据层和运维层进行了架构设计，采用负载均衡、健康检查、Orleans Actor、消息队列、限流、熔断、降级、MySQL 主从备份、幂等补偿和监控告警等措施，提高系统持续服务能力和故障恢复能力。实践表明，高可用架构不是简单增加服务器，而是围绕核心业务链路建立冗余、隔离、削峰、恢复和可观测能力的系统工程。

## 正文

### 一、项目背景

我参与设计并建设了一套面向移动游戏的高并发在线服务平台，项目客户端主要采用 Unity 和 Cocos 引擎开发，服务端为多款手机游戏提供统一的在线服务能力。平台主要包括玩家登录、角色管理、组队匹配、排行榜、商城支付、邮件发放、活动配置、数据统计和运营后台等模块。我在项目中主要负责服务端架构设计、核心链路改造和稳定性治理工作。

移动游戏业务具有明显的流量峰值特征。在新服开启、版本更新、节日活动和限时礼包上线时，大量玩家会集中登录、拉取配置、参与活动、支付购买和领取奖励。如果系统缺少高可用设计，单个服务实例异常、数据库压力过高、缓存失效或下游服务超时，都可能导致大面积请求失败，进而影响玩家体验和业务收入。

原有系统中部分业务采用同步调用方式，活动领奖、支付发货和日志记录等链路较长，高峰期容易出现接口响应慢、数据库写入压力大、故障定位困难等问题。因此，我们在平台建设中将高可用作为重要架构目标，通过服务拆分、冗余部署、流量削峰、故障隔离、数据补偿和监控告警等手段，提高系统在高并发和局部故障情况下的持续服务能力。

### 二、高可用架构设计目标

高可用架构的核心目标，是在局部故障、流量波动和外部依赖异常时，系统仍能尽量保持核心业务可用，并能够快速发现、定位和恢复问题。结合移动游戏业务特点，我们主要确定了以下设计目标。

第一，保证核心链路持续可用。登录、角色加载、支付下单、道具发放和活动参与是游戏平台的核心链路，必须优先保障。当系统资源紧张时，可以暂时降低排行榜刷新、非核心统计和部分运营展示能力，但不能轻易影响玩家进入游戏和关键交易。

第二，避免局部故障扩散。一个玩家状态异常、一个房间服务异常、一个活动服务实例故障，都不应导致整个平台不可用。因此系统需要通过服务拆分、实例隔离、超时控制和熔断机制，限制故障影响范围。

第三，承受高峰流量冲击。开服和活动期间的请求量可能在短时间内迅速上升，如果所有请求直接进入后端服务和数据库，容易造成系统雪崩。因此需要通过限流、缓存、消息队列和削峰填谷手段，保护核心资源。

第四，保证关键数据可恢复。支付订单、道具发放、邮件奖励等业务关系到玩家资产，必须保证可追踪、可重试、可补偿。即使部分环节失败，也要能根据订单号、玩家 ID 和业务流水恢复处理。

第五，保证运行状态可观测。高可用不仅是故障时不宕机，还包括故障能够被及时发现、定位和处理。因此平台需要建设监控指标、日志追踪、告警规则和应急预案。

### 三、高可用架构的关键设计措施

首先，在接入层采用负载均衡和健康检查。玩家请求先进入网关和负载均衡层，再转发到后端服务实例。我们对登录、活动、支付等服务采用多实例部署，并通过健康检查发现异常实例。当某个实例响应超时或健康检查失败时，负载均衡将其从可用节点中摘除，避免请求继续打到异常实例上。这样既提高了系统吞吐能力，也消除了单实例故障带来的风险。

其次，在服务层采用 Orleans Actor 模型和服务拆分实现故障隔离。平台中玩家、房间、匹配队列等对象具有明显的状态特征，我们使用 Actor 对有状态对象进行建模，使不同玩家和房间的状态相互隔离。单个 Actor 出现异常时，影响范围通常限制在对应玩家或房间，不会扩散到整个服务。同时，我们将登录、活动、支付、邮件、排行榜等能力拆分为相对独立的服务，使模块之间保持高内聚、低耦合，便于独立扩容和故障处理。

第三，在流量层采用限流和削峰填谷。对于登录、活动领奖、排行榜查询等高频接口，我们在网关和服务端设置限流策略，按照用户、接口和时间窗口控制请求量，避免异常流量或重复点击拖垮后端。对于邮件发放、奖励到账、日志写入和部分统计任务，我们采用消息队列异步处理。请求进入系统后先完成资格校验和任务入队，再由后台消费者按照系统处理能力逐步消费，从而将瞬时高峰摊平，降低数据库和下游服务压力。

第四，在下游依赖调用中采用超时、重试、熔断和降级。支付确认、第三方渠道验证、排行榜服务和邮件服务都可能出现响应慢或短时不可用。我们为这些调用设置合理超时时间，避免线程长时间等待；对可重试的短暂失败进行有限次数重试；当某个下游服务持续异常时，启动熔断机制，短时间内停止继续调用，快速返回失败或降级结果。对于非核心功能，如排行榜展示、部分运营推荐和统计查询，可以返回缓存数据或默认数据，优先保障登录、支付和发奖等核心链路。

第五，在数据层采用 MySQL 主从、备份、幂等和补偿机制。平台的玩家账号、角色数据、订单和道具流水存储在 MySQL 中。为了提高数据可用性，我们对核心库进行主从部署和定期备份，并制定恢复预案。对于支付和道具发放链路，我们设计了订单表、发放流水表和补偿状态字段，通过订单号和业务流水号保证幂等处理。即使支付回调、发奖或邮件发送某个环节失败，也可以通过后台补偿任务重新扫描并处理未完成记录，避免重复发放或漏发。

第六，在运维层建立监控告警、日志追踪和灰度发布机制。我们对接口 QPS、响应时间、错误率、服务实例状态、队列积压、数据库连接数和慢 SQL 等指标进行监控，并为核心指标设置告警规则。对于支付和发奖链路，日志中记录玩家 ID、订单号、请求链路和处理状态，方便故障后快速定位。版本发布时，我们采用灰度发布方式，先在少量服务器或内部账号验证，观察监控指标正常后再逐步扩大范围，降低发布变更带来的可用性风险。

### 四、实施过程中的问题和处理

在高可用改造过程中，我们也遇到了一些问题。首先是消息队列带来的重复消费问题。活动奖励和邮件发放改为异步处理后，网络抖动或消费者重启可能导致同一消息被重复投递。对此，我们在业务层引入幂等键，以玩家 ID、活动 ID 和奖励批次作为唯一约束，消费前先检查处理状态，确保重复消息不会造成重复发奖。

其次是支付和道具发放链路较长，故障定位比较困难。一次支付可能涉及客户端下单、渠道支付、服务端回调、订单确认、道具发放和邮件通知等多个环节。为解决这个问题，我们为每笔订单建立统一流水号，在各服务日志中传递该流水号，并在运营后台展示订单状态和补偿状态，使客服和运维人员能够快速判断问题出在哪个环节。

第三是缓存、数据库和异步任务之间存在一致性问题。部分活动数据和排行榜数据需要缓存来提高性能，但缓存更新不及时可能导致玩家看到旧数据。对此，我们采用缓存过期、主动刷新和后台修正相结合的方式，对于资产类强一致数据以数据库流水为准，对于排行榜等弱一致场景允许短时间延迟。

第四是服务拆分后链路变长，系统复杂度上升。为避免过度拆分带来维护成本，我们按照业务边界和可用性要求进行拆分，只对登录、活动、支付、邮件、排行榜等核心模块独立部署，对简单后台能力保持相对集中。同时补充接口文档、调用规范和监控看板，降低团队协作成本。

### 五、实施效果

通过高可用架构改造，平台在多次新服开启和限时活动中保持了较好的稳定性。活动高峰期间，接入层可以通过负载均衡将请求分散到多个服务实例，缓存和消息队列承担了削峰作用，核心接口响应更加稳定。单个服务实例异常时，健康检查能够将其摘除，故障影响范围明显缩小。

在支付和道具发放方面，订单流水、幂等处理和补偿机制提高了关键链路的可靠性。即使某些环节因网络或下游服务异常失败，也可以通过补偿任务继续处理，避免玩家资产丢失。监控告警和日志追踪也缩短了故障发现和定位时间，使运维和开发人员能够更快恢复服务。

同时，高可用设计也带来了额外复杂度。例如消息队列需要处理积压和重复消费，熔断降级需要选择合理阈值，数据库主从和缓存会引入一致性问题。因此，我们在实践中没有追求所有模块同等高可用，而是优先保障登录、支付、发奖和活动参与等核心链路，对非核心功能采取适度降级和延迟处理。

### 六、总结

高可用架构设计是保障在线系统稳定运行的重要能力。在本项目中，我们围绕面向移动游戏的高并发在线服务平台，从接入层、服务层、流量层、数据层和运维层设计了负载均衡、健康检查、Orleans Actor、限流、削峰填谷、熔断降级、MySQL 主从备份、幂等补偿和监控告警等措施。实践表明，高可用不是简单增加服务器，而是通过冗余、隔离、削峰、恢复和可观测性保证核心链路持续可用。作为架构设计人员，需要在可用性、复杂度、成本和一致性之间进行合理权衡，使架构真正服务于业务目标。
