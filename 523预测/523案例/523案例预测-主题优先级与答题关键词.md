### 3.3 架构组成模板

```text
接入层：Web、移动端、企业 IM、运营后台。
权限层：统一认证、角色权限、数据权限过滤。
知识处理层：文档解析、切分、清洗、元数据标注、向量化。
检索层：向量数据库、关键词检索、知识图谱检索、混合检索。
编排层：问题改写、检索结果重排、上下文拼接、Prompt 模板。
模型服务层：大模型推理服务、专有小模型、模型网关。
数据层：文档库、向量库、知识图库、业务数据库、日志库。
治理层：监控、审计、反馈、评测、版本管理。
```

### 3.4 RAG 流程模板

```text
1. 用户提交问题。
2. 系统进行身份认证和权限识别。
3. 对问题进行改写、意图识别和关键词提取。
4. 在向量库、关键词索引和知识图谱中进行混合检索。
5. 对检索结果按相关性、权限、时效性进行重排。
6. 将高质量片段、来源信息和约束条件拼接到 Prompt 中。
7. 调用大模型生成答案。
8. 返回答案、引用来源和置信提示。
9. 记录问题、检索内容、模型输出和用户反馈，用于评测优化。
```

### 3.5 幻觉治理模板

```text
1. 采用 RAG，让模型基于检索内容回答。
2. 要求答案必须给出引用来源。
3. 检索结果进行重排、去重和质量过滤。
4. 使用知识图谱和业务规则校验关键结论。
5. 缺少依据时要求模型回答无法确认。
6. 建立离线评测集，持续评估答案准确率和引用准确率。
```

### 3.6 防泄露模板

```text
1. 用户提问前进行身份认证和角色识别。
2. 文档按角色、部门、项目、设备、密级打元数据标签。
3. 检索前限定检索范围。
4. 检索后再次做权限过滤，禁止无权限片段进入 Prompt。
5. 敏感字段脱敏，传输和存储加密。
6. 模型调用经过统一网关并记录审计日志。
```

### 4.3 微服务改造模板

```text
1. 按业务能力拆分服务，例如账号、订单、支付、活动、排行榜、邮件、日志等。
2. 通过 API 网关统一接入，完成认证、鉴权、限流、路由和协议适配。
3. 通过服务注册与发现处理服务实例变化。
4. 通过配置中心统一管理配置和开关。
5. 通过限流、熔断、降级和超时控制提升韧性。
6. 通过日志、指标和链路追踪建设可观测体系。
7. 通过 CI/CD、灰度发布和自动化部署降低发布风险。
```

### 4.4 事件驱动模板

```text
1. 将业务事实抽象为事件，如支付成功、活动领奖、排行榜积分变化。
2. 核心服务只发布事件，不直接依赖所有下游服务。
3. 下游服务通过消息队列订阅事件，实现邮件通知、日志统计、数据分析等。
4. 使用本地消息表保证事件可靠投递。
5. 消费者通过幂等键、唯一约束和去重表保证幂等消费。
6. 对失败消息进行重试、死信队列和人工补偿。
7. 接受短暂最终一致，并通过状态查询和补偿保证最终正确。
```

### 4.5 可观测性模板

```text
日志：记录请求参数、错误码、traceId、关键业务状态。
指标：QPS、响应时间、错误率、CPU、内存、队列积压。
链路追踪：跟踪一次请求经过网关、服务、数据库、缓存、消息队列的完整链路。
告警：对错误率升高、响应变慢、队列积压和服务不可用进行告警。
```

### 5.3 Cache Aside 模板

```text
读流程：
1. 先读缓存。
2. 缓存命中则直接返回。
3. 缓存未命中则读数据库。
4. 将数据库结果写入缓存。
5. 返回结果。

写流程：
1. 先更新数据库。
2. 再删除缓存。
3. 后续读请求未命中缓存时重新加载数据库并回填缓存。
```

### 5.4 缓存异常模板

```text
缓存穿透：查询不存在的数据，绕过缓存打到数据库。
措施：布隆过滤器、空值缓存、参数校验。

缓存击穿：热点 Key 过期，大量请求同时打到数据库。
措施：互斥锁、热点 Key 不过期、逻辑过期、提前刷新。

缓存雪崩：大量 Key 同时过期或 Redis 故障。
措施：过期时间随机化、缓存预热、Redis 集群、限流降级。
```

### 5.5 数据库扩展模板

```text
读写分离：主库写，从库读，通过主从复制同步数据。
水平分区：按行拆分，适合数据量大。
垂直分区：按列或业务拆分，适合字段多、冷热字段差异大。
范围分区：按时间、区域等范围划分。
哈希分区：按哈希函数划分，数据分布较均匀。
索引优化：为高频查询字段建索引，避免索引过多影响写入。
```


### 6.5 风险点/敏感点/权衡点模板

```text
风险点：当前架构中可能导致质量属性无法满足的设计问题。
敏感点：某个架构决策变化会显著影响质量属性。
权衡点：一个设计决策同时影响多个质量属性，提升一个可能损害另一个。
```

例子：

```text
使用缓存可以提升性能，但会带来数据一致性风险，这是权衡点。
网关限流阈值会直接影响系统可用性和用户体验，这是敏感点。
所有服务共用一个数据库，可能导致扩展性和故障隔离不足，这是风险点。
```

### 7.2 DFD 必背

```text
外部实体：系统之外的人、组织、系统。
加工：对数据进行处理的过程，通常用动宾短语。
数据存储：系统内部保存的数据。
数据流：数据在实体、加工、存储之间的流动。
```

DFD 平衡：

```text
父图和子图的输入输出数据流必须保持一致。
加工必须有输入也有输出。
数据存储不能直接连外部实体。
外部实体之间不能直接连数据流。
数据流名称应是数据名，不是动作名。
```

### 7.3 UML 必背

```text
用例图：参与者、用例、包含 include、扩展 extend、泛化。
类图：类、属性、方法、关联、聚合、组合、继承、依赖。
顺序图：对象生命线、消息、返回消息、激活。
状态图：状态、事件、迁移、初态、终态。
活动图：活动、判断、分支、合并、并发。
部署图：节点、构件、通信路径。
```

## 9. 考场做题顺序建议

拿到题后先用 3 分钟判断题型。

优先选择：

```text
1. 大模型/RAG/知识图谱：如果题目材料足够清楚，优先做。
2. 数据缓存高并发：有套路，容易拿步骤分。
3. 质量属性架构评估：术语准确就能拿分。
4. 微服务云原生：看题目是否有明确问法。
5. DFD/UML：如果图清晰、空不多，可做；如果图复杂，谨慎。
6. 嵌入式/AUTOSAR：熟悉才做，不熟就作为备选。
```

答题时不要空泛，尽量写成：

```text
措施 + 解决的问题 + 适用原因 + 风险或权衡
```
