## 题目原文

```text
第5题：Web架构
```

### 问题原文

【问题 1】（11 分）

数据库类型选择，从下列选择中填入（ ）到（ ）。

选项如下：

```text
（A）处理层
（B）接口层
（C）存储层
（D）HBase
（E）HDFS
（F）ES
（G）MySQL
```

【问题 2】（10 分）

MongoDB 特点。

```text
2.1 说明 MongoDB 如何实现空间矢量化存储的。
2.2 说明 MongoDB 对于存储非关系型数据的优势。
```

【问题 3】（4 分）

冷热数据。

```text
请说明 HDFS 使用热数据、温数据和冷数据存储的原因。
```

```text
（5）D HBase

```

### 问题 2 答案

MongoDB 实现空间矢量化存储：

```text
MongoDB 可使用 GeoJSON 格式存储点、线、面、多边形等空间对象，
并通过 2dsphere 等空间索引支持空间范围查询、相交查询、邻近查询等操作。
```

MongoDB 存储非关系型数据的优势：

```text
MongoDB 采用文档模型，使用 BSON/JSON 形式保存数据，Schema 灵活，
支持嵌套对象和数组，适合存储结构变化频繁、属性差异较大、半结构化或非结构化数据。
```

### 问题 3 答案

HDFS 使用热数据、温数据、冷数据分层存储的原因：

```text
根据数据访问频率和价值进行分层管理，在性能和成本之间取得平衡。
热数据访问频繁，应放在高性能存储并保持较高访问效率；
温数据访问频率中等，可采用普通存储；
冷数据访问很少，适合放在低成本归档存储，降低 PB 级数据存储成本。
```

## 问题 1 解题过程

### 1. 判断三层架构

图中最上层包含：

```text
Client
REST API
Web页面
```

这些组件直接面向用户或外部系统，负责提供访问入口和接口。

所以：

```text
（1）接口层 -> B
```

图中中间层包含：

```text
队列任务
原始数据入库
瓦片生成
矢量要素入库
元数据生成
索引构建
```

这些组件负责异步任务调度、数据加工、入库和索引构建，属于业务处理或数据处理过程。

所以：

```text
（2）处理层 -> A
```

图中最下层是多个数据库和存储系统，负责保存不同类型的数据。

所以：

```text
（3）存储层 -> C
```

### 2. 判断存储系统

#### （4）原始数据入库

原始空间数据、日志、文件和 PB 级大数据更适合放入分布式文件系统。

HDFS 适合：

```text
海量文件存储
低成本扩展
大规模离线分析
原始数据归档
```

所以：

```text
（4）HDFS -> E
```

#### （5）瓦片生成

地图瓦片通常具有：

```text
数量巨大
按层级、行列号快速访问
读多写少
Key-Value 特征明显
```

HBase 适合海量稀疏表和按 RowKey 快速访问的场景，常用于存储地图瓦片、时间序列或宽表数据。

所以：

```text
（5）HBase -> D
```

#### （6）矢量要素或结构化业务数据入库

图中 MongoDB 已作为固定的空间文档/矢量存储出现，剩余选项中 MySQL 适合保存结构化元数据、任务状态、数据目录、关系型业务表等。

结合图中数据处理链路，空（6）可填：

```text
（6）MySQL -> G
```

说明：

```text
如果题目语境强调“矢量要素以 GeoJSON 文档存储”，则 MongoDB 是空间矢量数据的核心存储；
但本题选项中 MongoDB 已在图中给出，不在备选项内，所以空（6）从备选项中对应 MySQL。
```

#### （7）索引构建

索引构建通常对应搜索引擎。

ES，也就是 Elasticsearch，适合：

```text
全文检索
倒排索引
复杂查询
空间检索
日志检索
快速搜索
```

所以：

```text
（7）ES -> F
```

## 问题 2 解题过程

### 2.1 MongoDB 如何实现空间矢量化存储

MongoDB 可以使用 GeoJSON 格式存储空间矢量对象。

常见空间对象包括：

```text
Point 点
LineString 线
Polygon 面
MultiPoint 多点
MultiLineString 多线
MultiPolygon 多面
```

示例结构：

```json
{
  "name": "road_001",
  "type": "road",
  "geometry": {
    "type": "LineString",
    "coordinates": [
      [116.30, 39.90],
      [116.31, 39.91]
    ]
  },
  "properties": {
    "level": "main",
    "width": 12
  }
}
```

MongoDB 可对空间字段建立空间索引：

```text
2dsphere index
```

支持的典型查询：

```text
附近查询
范围查询
相交查询
包含查询
空间邻近分析
```

答题模板：

```text
MongoDB 通过 GeoJSON 文档模型保存点、线、面等空间矢量对象，
并通过 2dsphere 空间索引支持空间范围、相交和邻近查询，
因此可以较自然地表达和检索空间矢量数据。
```

### 2.2 MongoDB 存储非关系型数据的优势

MongoDB 属于文档数据库，适合半结构化和非结构化数据。

优势包括：

```text
Schema 灵活，不要求所有记录字段完全一致
支持 BSON/JSON 文档，便于表达复杂对象
支持嵌套对象和数组，减少多表 JOIN
适合属性变化频繁的数据
支持水平扩展和分片
支持高并发读写
适合日志、空间要素、内容、配置、元数据等场景
```

答题模板：

```text
MongoDB 采用文档模型，能够以 BSON/JSON 形式保存结构灵活的数据，
支持嵌套对象和数组，避免关系数据库中大量表拆分和关联查询。
对于空间要素、评论、元数据等字段差异大、结构变化快的数据，
MongoDB 更容易扩展字段并支持水平扩展，因此适合存储非关系型数据。
```

## 问题 3 解题过程

### 1. 冷热数据分层的本质

冷热数据分层的核心是：

```text
按访问频率、业务价值和性能要求选择不同存储策略。
```

在 PB 级数据场景中，如果所有数据都使用高性能、高副本、高成本存储，会导致成本过高。

### 2. 热数据

热数据特点：

```text
访问频繁
近期数据
查询延迟要求高
业务价值高
```

处理方式：

```text
放在高性能存储
保留较高副本
支持快速查询和计算
```

例子：

```text
最近几天的用户访问日志、热门地图瓦片、近期矢量要素。
```

### 3. 温数据

温数据特点：

```text
访问频率中等
仍可能被分析使用
性能要求低于热数据
```

处理方式：

```text
放在普通磁盘或标准 HDFS 存储
保持成本和访问效率平衡
```

例子：

```text
最近几个月的历史行为数据、较少访问的瓦片数据。
```

### 4. 冷数据

冷数据特点：

```text
访问很少
主要用于归档、审计、离线训练或历史追溯
数据量大
成本敏感
```

处理方式：

```text
放在低成本存储
降低副本成本或使用归档策略
必要时压缩存储
```

例子：

```text
多年历史日志、旧版本空间数据、过期统计数据。
```

### 5. 本题答法

```text
HDFS 对热数据、温数据和冷数据进行分层存储，是为了根据访问频率和业务价值采用不同的存储策略。
热数据访问频繁，需要较高性能和较快响应；
温数据访问频率中等，可采用普通存储以兼顾性能和成本；
冷数据访问很少，适合低成本归档或压缩存储。
这样可以在 PB 级数据规模下，既保证热点数据访问性能，又降低整体存储成本。
```

## 标准答题模板

### 数据库类型选择模板

```text
接口层负责对外提供 Client、REST API 和 Web 页面访问；
处理层负责队列任务、数据加工、瓦片生成、矢量入库、元数据生成和索引构建；
存储层负责使用不同数据库保存不同类型的数据。
HDFS 适合原始大文件和海量数据存储；
HBase 适合海量瓦片或宽表 Key-Value 数据；
MySQL 适合结构化业务数据和元数据；
ES 适合索引构建和全文检索。
```

### MongoDB 模板

```text
MongoDB 使用文档模型保存数据，可通过 GeoJSON 表达点、线、面等空间矢量对象，
并使用 2dsphere 空间索引支持空间查询。
它的 Schema 灵活，支持嵌套对象和数组，适合存储结构变化快、字段差异大、
半结构化和非结构化的数据。
```

### 冷热数据模板

```text
冷热数据分层是为了在性能和成本之间取得平衡。
热数据访问频繁，应使用高性能存储；
温数据访问频率中等，采用普通存储；
冷数据访问很少，采用低成本归档或压缩存储。
这样可以降低 PB 级数据的整体存储成本，同时保障热点数据访问效率。
```

## 关键词速记

```text
接口层：Client、REST API、Web 页面
处理层：队列任务、数据加工、瓦片生成、索引构建
存储层：HDFS、HBase、MongoDB、MySQL、ES
HDFS：原始数据、大文件、低成本、分布式存储
HBase：海量瓦片、宽表、RowKey、Key-Value
MongoDB：GeoJSON、2dsphere、文档模型、Schema 灵活
MySQL：结构化数据、元数据、关系型数据
ES：全文检索、倒排索引、搜索、索引构建
冷热数据：访问频率、性能、成本、归档
```

## 易错点

1. 不要把 REST API 所在层填成处理层，它属于对外接口层。
2. 队列任务和瓦片生成属于处理层，不是存储层。
3. HDFS 是分布式文件系统，适合原始大数据存储，不是搜索引擎。
4. ES 主要用于索引和搜索，不适合作为原始文件主存储。
5. MongoDB 不是关系型数据库，它适合文档型、半结构化、空间矢量数据。
6. 冷热数据分层的核心不是分类好看，而是平衡性能和成本。
