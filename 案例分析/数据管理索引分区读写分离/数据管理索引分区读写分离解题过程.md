# 数据管理索引分区读写分离解题过程

## 题目原文

### 试题信息

```text
试题四（25分）

阅读以下关于数据管理的叙述，在答题纸上回答问题1至问题3。
```

### 说明原文

某全国连锁药店企业在新冠肺炎疫情期间，紧急推出在线口罩预约业务系统。该业务系统为普通用户提供口罩商品查询、购买、订单查询等业务，为后台管理人员提供订单查询、订单地点分布汇总、物流调度等功能。该系统核心的关系模式为预约订单信息表。

推出业务系统后，几天内业务迅速增长到每日10万多笔预约订单，系统数据库服务器压力剧增，导致该业务交易响应速度迅速降低，甚至出现部分用户页面无法刷新、预约订单服务无响应的情况。为此，该企业紧急成立技术团队，由张工负责，以期尽快解决该问题。

### 问题原文

#### 问题1（9分）

经过分析，张工认为当前预约订单信息表存储了所有订单信息，记录已达到了百万级别。系统主要的核心功能均涉及对订单信息表的操作，应首先优化预约订单信息表的读写性能，建议针对系统中的SQL语句，建立相应索引，并进行适当的索引优化。

针对张工的方案，其他设计人员提出了一些异议，认为索引过多有很多副作用。请用100字以内的文字简要说明索引过多的副作用。

#### 问题2（10分）

作为团队成员之一，李工认为增加索引并进行优化并不能解决当前问题，建议采用物理分区策略，可以根据预约订单信息表中“所在城市”属性进行表分区，并将每个分区分布到独立的物理磁盘上，以提高读写性能。常见的物理分区特征如表4-1所示。李工建议选择物理分区中的列表分区模式。

请填补表4-1中的空（a）--（d）处，并用100字以内的文字解释说明李工选择该方案的原因。

#### 问题3（6分）

在系统运行过程中，李工发现后台管理人员执行的订单地址信息汇总等操作，经常出现与普通用户的预约订单操作形成读写冲突，影响系统的性能。因此李工建议采用读写分离模式，采用两台数据库服务器，并采用主从复制的方式进行数据同步。请用200字以内的文字简要说明主从复制的三种模式。

## 解题过程

## 问题 1：索引过多的副作用

索引的作用是加快查询，但索引不是越多越好。

原因是：

```text
索引本身也是数据库对象，需要占用磁盘空间。
```

同时，对表进行写操作时，数据库不仅要修改数据页，还要维护相关索引。

因此：

```text
INSERT 要插入索引项
DELETE 要删除索引项
UPDATE 若修改索引列，还要更新索引项
```

所以索引过多会导致：

- 存储空间增加。
- 写操作变慢。
- 索引维护成本上升。
- 统计信息和优化器选择成本增加。
- 备份、恢复、重建索引等运维成本增加。

本题要求 100 字以内，可以答：

```text
索引会占用额外存储空间；插入、删除和更新数据时需要同步维护索引，降低写入性能；索引过多还会增加优化器选择成本和数据库维护成本。
```

## 问题 2：物理分区模式填空

常见物理分区包括：

```text
范围分区
哈希分区
列表分区
组合分区
```

### 2.1 范围分区

范围分区依据属性取值范围划分数据。

例如：

```text
按订单日期分区
按订单编号范围分区
按金额范围分区
```

因此，范围分区适合：

```text
连续/范围型数据
```

即：

```text
(b)：连续/范围型数据
```

范围分区的管理能力强，因为可以按时间或范围批量归档、删除、迁移数据。

但如果某些范围内数据量特别大，就会产生数据倾斜，所以数据分布可能不均匀。

### 2.2 哈希分区

哈希分区依据属性的哈希函数值划分数据。

例如：

```text
hash(order_id) % N
```

因此：

```text
(a)：属性的哈希函数值
```

哈希分区通常可以让数据分布更均匀，但不利于按业务范围直接管理数据，所以数据管理能力较弱。

### 2.3 列表分区

列表分区依据属性的离散值划分数据。

例如：

```text
城市 = 北京 -> 分区 P_BEIJING
城市 = 上海 -> 分区 P_SHANGHAI
城市 = 广州 -> 分区 P_GUANGZHOU
```

列表分区适合：

```text
单属性、离散值
```

但不同城市订单量可能差异较大，所以列表分区的数据分布通常：

```text
(d)：不均匀
```

### 2.4 组合分区

组合分区是多种分区方式的组合。

例如：

```text
先按日期范围分区，再按城市列表分区
先按城市列表分区，再按订单号哈希分区
```

组合分区可以兼顾范围管理和数据分布，因此数据管理能力通常：

```text
(c)：能力强
```

## 问题 2：为什么选择列表分区

本题预约订单信息表中有“所在城市”属性。

“所在城市”的特点是：

```text
单个属性
取值离散
业务上常按城市查询、汇总和调度
```

例如：

```text
北京、上海、广州、深圳、成都、武汉
```

这些取值不是连续范围，也不适合用哈希后隐藏业务含义。

后台管理人员需要：

- 订单查询。
- 订单地点分布汇总。
- 物流调度。

这些功能天然和城市维度相关。

因此，使用列表分区可以：

- 按城市直接定位分区。
- 减少全表扫描。
- 将不同城市分区放到不同磁盘，降低 I/O 竞争。
- 便于城市维度统计、汇总和调度。

100 字以内答法：

```text
“所在城市”是典型的单属性离散值，适合按城市列表进行分区。各城市订单可映射到不同分区，并分布到独立物理磁盘上，便于按城市查询、汇总和调度，减少单表访问压力，提高读写性能。
```

## 问题 3：主从复制三种模式

读写分离通常采用：

```text
主库负责写操作
从库负责读操作
主从之间通过复制同步数据
```

主从复制按同步确认机制常分为三种模式。

### 3.1 异步复制

主库执行事务并提交后，不等待从库确认，直接向客户端返回成功。

从库随后异步接收主库日志并重放。

特点：

```text
性能最好
主从延迟可能较大
主库故障时可能丢失尚未同步到从库的数据
```

### 3.2 同步复制

主库提交事务时，需要等待从库确认数据写入成功后，才向客户端返回成功。

特点：

```text
一致性最好
响应延迟较高
主库容易受从库性能影响
```

### 3.3 半同步复制

主库提交事务后，至少等待一个从库确认收到日志，再向客户端返回成功。

特点：

```text
一致性强于异步复制
性能优于完全同步复制
是性能和可靠性之间的折中方案
```

200 字以内答法：

```text
异步复制：主库提交事务后立即返回，从库异步拉取或接收日志并重放，性能好但可能存在延迟和数据丢失风险。同步复制：主库需等待从库确认数据写入后再提交，数据一致性强，但响应时间长、性能开销大。半同步复制：主库至少等待一个从库确认收到日志后再返回，是性能与一致性之间的折中方案。
```

## 最终答案汇总

```text
问题1：
索引会占用额外存储空间；插入、删除和更新数据时需要同步维护索引，降低写入性能；索引过多还会增加优化器选择成本和数据库维护成本。

问题2：
(a) 属性的哈希函数值
(b) 连续/范围型数据
(c) 能力强
(d) 不均匀

选择列表分区原因：
“所在城市”是典型的单属性离散值，适合按城市列表进行分区。各城市订单可映射到不同分区，并分布到独立物理磁盘上，便于按城市查询、汇总和调度，减少单表访问压力，提高读写性能。

问题3：
异步复制、同步复制、半同步复制。
```

## 易错点

1. 索引能提高查询性能，但会降低写入性能，不是越多越好。
2. “所在城市”是离散值，不是连续范围，因此更适合列表分区。
3. 哈希分区数据分布均匀，但业务管理能力弱。
4. 列表分区便于按业务值管理，但可能因城市订单量不同导致数据不均匀。
5. 主从复制三种模式通常指异步复制、同步复制和半同步复制。
