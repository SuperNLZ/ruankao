# AUTOSAR 汽车电子基础软件架构-快速记忆

## 0. 题目原文

试题三，25 分。

阅读以下关于汽车电子基础软件架构的说明，在对应栏内回答问题 1 至问题 3。

说明：

某软件公司承担了某项国家重点项目的研制工作，任务重点是参考国外汽车电子发展趋势，开发某汽车公司的企业汽车电子基础软件的架构，逐步实现汽车企业未来技术发展规划。

该软件公司接受此项任务后，调动全体技术人员深入收集国外相关技术资料，经过多方调研和分析，公司提出遵照国际组织最新推出的 AUTOSAR 规范，按统筹规划、分步骤实施的原则，实现汽车公司的基础软件架构设计。图 3-1 给出了 AUTOSAR 规范所定义的工作包，图中灰色部分代表本项目工作所包含的内容，即软件架构和基础软件。

问题 1，9 分：

AUTOSAR 规范中要求，汽车电子软件开发流程应尽量满足并发、可多次迭代的特性。为了定义汽车电子的软件开发过程，公司李工和王工分别提出了两种软件开发流程，其开发流程见图 3-2 和图 3-3，图中 ECU 是指汽车电子中的电子控制单元。请说明李工和王工谁定义的流程更符合 AUTOSAR 的规定，并说明理由。

## 2. 问题 1 答案

结论：

```text
李工定义的流程更符合 AUTOSAR 的规定。
```

理由：

```text
AUTOSAR 方法强调软件组件描述、系统描述和 ECU 资源描述可以并行开展，
再通过系统配置生成 ECU 相关描述，最后进行各 ECU 配置和可执行软件生成。
李工流程体现了并发和迭代，王工流程把这些活动串行化，不符合 AUTOSAR 的并发迭代思想。
```

答题模板：

```text
李工定义的流程更符合 AUTOSAR 的规定。
原因是 AUTOSAR 的开发流程强调并发、迭代以及软件与硬件 ECU 的解耦。
在 AUTOSAR 中，软件组件描述、系统描述和 ECU 资源描述可以根据需求和车辆信息并行开展；
之后通过系统配置完成软件组件到 ECU 的映射，并生成各 ECU 所需的描述信息；
再针对每个 ECU 进行配置，最终生成 ECU 可执行软件。
王工流程将相关活动顺序串行化，且软件组件描述和 ECU 资源描述的位置不合理，
不利于并发开发和多轮迭代，因此不如李工流程符合 AUTOSAR 规定。
```

## 3. 问题 2 答案

图 3-4 中 1 至 5 箭头含义如下：

```text
1：操作系统的警报管理发现 ECU 系统出错时，启动错误处理程序。
2：错误处理程序将具体动作交给任务/进程管理模块，对发生错误的任务进行处理。
3：错误处理程序产生一个错误事件。
4：任务控制块处理程序调用资源管理功能，实现硬件资源重新分配。
5：任务控制块处理程序通知事件管理模块，对错误事件进行应答。
```

可压缩成表格记忆：

| 箭头 | 含义 |
|---|---|
| 1 | 警报管理发现错误，启动错误处理 |
| 2 | 错误处理交给任务/进程管理处理故障任务 |
| 3 | 错误处理产生错误事件 |
| 4 | 任务控制块调用资源管理，重新分配资源 |
| 5 | 任务控制块通知事件管理，应答错误事件 |

## 4. 问题 3 答案

150 字以内答案：

```text
采用 AUTOSAR 开放式架构，可统一汽车电子基础软件接口和开发方法，支持不同厂商协同开发，降低软硬件耦合，提高软件复用性、可移植性和系统可维护性，并可通过配置生成 ECU 软件，便于分阶段实施和持续演进。汽车电子 ECU 主要覆盖动力、底盘和车身三个领域。
```

关键词：

```text
开放标准
统一接口
软硬件解耦
复用性
可移植性
可维护性
配置生成
动力、底盘、车身
```

## 5. 核心速记

```text
问题 1：选李工，理由是并发、迭代、软件组件与 ECU 解耦。
问题 2：警报发现错误 -> 错误处理 -> 任务处理/错误事件 -> 资源重分配/事件应答。
问题 3：优点写开放标准、统一接口、降低耦合、提高复用和可移植；ECU 三领域写动力、底盘、车身。
```

## 6. 易错点

1. 问题 1 不要只说“李工”，必须补充“并发、迭代、软件与 ECU 解耦”。
2. 王工流程的问题不是图画得少，而是流程过于串行，软件组件和 ECU 资源描述位置不合理。
3. 问题 2 的箭头要围绕“警报、错误处理、任务管理、资源管理、事件管理”答。
4. 问题 3 要控制在 150 字以内，别写成长篇架构说明。
5. ECU 覆盖领域不要写成“发动机、刹车、车灯”等部件，应概括为“动力、底盘、车身”。
