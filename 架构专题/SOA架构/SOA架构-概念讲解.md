# SOA架构-概念讲解

## 1. 先说明

你这次问的是 SOA 架构。你原问题第 1 点写成了“理解云原生架构是什么”，这里按你的标题修正为：

```text
1. 理解 SOA 架构是什么
2. 掌握 SOA 的常见考点
```

## 2. SOA 一句话理解

SOA 的全称是：

```text
Service-Oriented Architecture
面向服务架构
```

一句话理解：

```text
SOA = 把企业中分散的系统能力封装成标准服务，通过统一接口、契约、注册、发现、调用和编排，实现系统集成与业务复用。
```

更考试化的定义：

```text
SOA 是一种组件模型，它将应用程序的不同功能单元封装为服务，
通过服务之间定义良好的接口和契约联系起来。
这些接口采用中立方式定义，独立于硬件平台、操作系统和编程语言，
使不同系统中的服务能够以统一、通用的方式进行交互。
```

最核心的理解：

```text
SOA 解决的核心问题不是“单个系统怎么写代码”，
而是“企业里多个异构系统如何低耦合地集成、复用和编排”。
```

## 3. 为什么会出现 SOA

企业信息化通常不是一次建成的，而是多年积累出来的。

一个企业里可能同时存在：

- 财务系统。
- 人事系统。
- 进销存系统。
- 客户关系管理系统。
- 订单系统。
- 报表系统。
- 老旧主机系统。
- 新建 Web 系统。

这些系统可能由不同厂商开发，运行在不同平台上，使用不同语言和数据库。

如果系统之间直接互相调用，就会形成大量点对点连接：

```text
系统 A 调系统 B
系统 B 调系统 C
系统 C 又调系统 A
系统 D 同时调 A、B、C
```

问题会越来越严重：

- 接口混乱。
- 重复开发。
- 数据不一致。
- 系统耦合严重。
- 一个系统变更会影响很多系统。
- 业务流程跨系统时难以自动化。
- 新业务上线需要大量改造旧系统。

SOA 就是为了解决这些问题。

SOA 的思路是：

```text
把各系统中有业务价值的能力抽象成服务；
用标准接口对外暴露；
通过服务注册、查找、调用、编排和治理，
把多个服务组合成新的业务流程。
```

## 4. SOA 中的“服务”到底是什么

你不要把 SOA 中的服务理解成一个普通函数，也不要简单理解成一个进程。

在 SOA 语境下，服务是：

```text
一个对外提供完整业务能力的功能单元。
```

例如：

| 服务 | 含义 |
|---|---|
| 查询客户信息服务 | 对外提供客户信息查询能力 |
| 创建订单服务 | 对外提供订单创建能力 |
| 财务记账服务 | 对外提供记账能力 |
| 支付确认服务 | 对外提供支付状态确认能力 |
| 航班信息查询服务 | 对外提供航班业务对象查询能力 |

教材中特别强调：

```text
SOA 把一个由多层服务组成的独立应用看作一个整体服务。
服务不是狭义的后台进程，而是对外提供一组完整功能的独立应用能力。
```

也就是说，财务系统本身可能是三层架构、数据库、消息队列和多个子模块组成的系统，但从企业整体架构看，它可以对外提供“财务处理服务”。

## 5. SOA 的基本角色

SOA 通常可以从三个角色理解。

### 5.1 服务提供者

服务提供者负责实现并发布服务。

例如：

```text
财务系统提供“付款确认服务”
库存系统提供“库存查询服务”
客户系统提供“客户信息服务”
```

### 5.2 服务消费者

服务消费者负责查找并调用服务。

例如：

```text
订单系统调用库存查询服务
结算系统调用付款确认服务
客服系统调用客户信息服务
```

### 5.3 服务注册中心

服务注册中心保存服务的描述信息，帮助消费者发现服务。

典型过程：

```text
发布 Publish：服务提供者把服务发布到注册中心。
查找 Find：服务消费者在注册中心查找服务。
绑定 Bind：服务消费者根据服务契约调用服务。
```

这个过程是 SOA 和 Web Service 题目中常考的基础模型。

## 6. SOA 的核心元素

### 6.1 服务接口

服务接口说明服务对外提供什么能力。

例如：

```text
createOrder()
queryCustomer()
confirmPayment()
getFlightInfo()
```

考试重点：

```text
服务接口应明确定义，并把公共接口与内部实现隔离开。
```

### 6.2 服务契约

服务契约说明服务消费者如何调用服务。

它通常包括：

- 服务名称。
- 输入参数。
- 输出结果。
- 数据格式。
- 调用协议。
- 服务地址。
- 安全策略。
- 服务质量要求。

在 Web Service 中，常用 WSDL 描述服务契约。

### 6.3 消息

SOA 服务之间通常通过消息通信。

消息可以是：

```text
XML
JSON
SOAP 消息
业务事件
请求/响应报文
```

SOA 强调服务之间通过标准消息交互，减少对具体语言、平台和实现的依赖。

### 6.4 服务注册与发现

服务注册与发现用于解决：

```text
服务在哪里？
服务能做什么？
如何调用服务？
```

Web Service 中常考：

```text
UDDI 用于服务描述、发现和集成。
```

### 6.5 企业服务总线 ESB

ESB 是 SOA 中非常重要的基础设施。

ESB 的作用可以理解为：

```text
企业内各服务之间的统一通信、路由、转换和集成通道。
```

它常见能力包括：

- 消息路由。
- 地址透明。
- 服务注册和命名管理。
- 请求/响应。
- 发布/订阅。
- 多协议支持。
- 数据格式转换。
- 协议转换。
- 日志和监控。
- 安全控制。

考试记忆：

```text
ESB = 连接服务的总线，负责路由、协议转换、数据转换、服务注册、消息传递、监控。
```

### 6.6 服务编排

服务编排是把多个服务组合成一个完整业务流程。

例如：

```text
下单流程 =
查询客户信息
+ 校验库存
+ 创建订单
+ 冻结库存
+ 调用支付
+ 通知发货
```

教材中会提到 BPEL：

```text
BPEL 用于组合、编排和协调 Web 服务，
把多个已有服务按业务流程组织成一个新的服务。
```

考试中看到：

```text
BPEL
业务流程执行语言
流程编排
组合 Web Service
```

就要想到 SOA 服务编排。

## 7. SOA 与 Web Service 的关系

这两个概念容易混淆。

结论：

```text
SOA 是架构思想；
Web Service 是实现 SOA 的一种常见技术方式。
```

SOA 可以使用 Web Service 实现，也可以使用消息中间件、交易中间件、REST、RPC、ESB 等方式实现。

Web Service 常用协议和规范：

| 名称 | 作用 |
|---|---|
| SOAP | 基于 XML 的消息交换协议 |
| WSDL | 描述服务做什么、如何访问、位于何处 |
| UDDI | 服务描述、发现和集成 |
| XML Schema / XSD | 定义消息格式和数据结构 |

### 7.1 SOAP

SOAP 是简单对象访问协议，用于在分布式环境中交换信息。

考试关键词：

```text
SOAP
XML
消息交换
分布式环境
```

### 7.2 WSDL

WSDL 是 Web 服务描述语言。

它描述 Web 服务的三个基本属性：

```text
服务做些什么：服务提供的操作或方法。
如何访问服务：交互数据格式和必要协议。
服务位于何处：协议相关地址，如 URL。
```

这三个点是选择题高频考点。

### 7.3 UDDI

UDDI 是统一描述、发现和集成协议。

它的作用：

```text
让服务提供者发布服务；
让服务消费者发现服务；
让双方根据服务描述进行集成。
```

记忆：

```text
UDDI = 服务注册和发现。
```

### 7.4 REST

REST 更常见于现代接口和微服务。

REST 的核心是把网络中的对象看作资源，通过统一接口访问资源。

考试关键词：

```text
资源
表述
状态转移
HTTP
RESTful API
```

SOA 早期常和 SOAP、WSDL、UDDI 绑定出现；现代系统中也可能用 REST 风格暴露服务。

## 8. SOA 参考架构

教材中提到 IBM WebSphere 业务集成参考架构，是典型的以服务为中心的企业集成架构。

它可以划分为六类服务。

### 8.1 业务逻辑服务

用于实现业务逻辑。

包括：

- 业务应用服务。
- 业务伙伴服务。
- 应用和信息资产。

### 8.2 控制服务

用于实现人、流程和信息的集成。

关键词：

```text
People
Process
Information
流程控制
业务编排
```

### 8.3 连接服务

通过企业服务总线实现服务之间的连接。

关键词：

```text
Connectivity Service
ESB
服务连接
消息路由
协议转换
```

### 8.4 业务创新和优化服务

用于监控业务系统运行时性能，并根据业务变化优化系统。

关键词：

```text
业务监控
业务性能
优化
适应变化
```

### 8.5 开发服务

提供贯穿软件开发生命周期的工具支持。

包括：

- 需求分析。
- 建模。
- 设计。
- 开发。
- 测试。
- 维护。

### 8.6 IT 服务管理

支持业务系统运行的基础设施管理能力。

包括：

- 安全服务。
- 目录服务。
- 系统管理。
- 资源虚拟化。

选择题常见陷阱：

```text
SOA 参考架构包括业务逻辑服务、控制服务、连接服务、业务创新和优化服务、开发服务、IT 服务管理。
“中间件服务”不是教材列出的六类之一。
```

## 9. SOA 的设计标准要求

### 9.1 文档标准化

SOA 服务应具有平台独立的自我描述文档。

Web Service 中常用 WSDL 描述服务。

考试关键词：

```text
WSDL
服务描述
平台独立
自我描述
```

### 9.2 通信协议标准

SOA 服务通过消息通信，消息通常由 XML Schema / XSD 定义。

考试关键词：

```text
消息通信
XML Schema
XSD
标准协议
```

### 9.3 统一登记与集成

SOA 服务通过注册中心维护。

消费者在注册中心查找服务并调用服务。

考试关键词：

```text
Registry
UDDI
服务登记
服务发现
```

### 9.4 服务质量 QoS

SOA 不只关心“能不能调通”，也关心服务质量。

QoS 常包括：

| 项目 | 含义 |
|---|---|
| 可靠性 | 消息传输特性，如仅传一次、最多传一次、重复过滤、保证传送 |
| 安全性 | 认证、授权、消息安全 |
| 策略 | 服务提供者要求消费者满足某种调用策略 |
| 控制 | 使用 BPEL 等控制服务组合和流程 |
| 管理 | 对运行在不同环境下的服务进行统一管理 |

## 10. SOA 的设计原则

这些原则是软考选择题高频内容。

### 10.1 无状态

服务不应该依赖上一次请求的状态。

为什么：

- 便于扩展。
- 便于负载均衡。
- 便于故障恢复。
- 降低服务之间的上下文依赖。

考试表达：

```text
SOA 服务应是独立的、自包含的请求，不依赖其他服务的上下文和状态。
```

### 10.2 单一实例

以高内聚方式实现服务，避免功能冗余。

简单理解：

```text
同一个业务能力不要在多个系统中重复实现。
```

### 10.3 明确定义的接口

服务接口要清晰定义。

服务消费者只依赖接口，不依赖内部实现。

考试关键词：

```text
接口契约
WSDL
公共接口
内部实现隔离
```

### 10.4 自包含和模块化

服务封装稳定、重复出现的业务活动。

服务实现实体应尽量：

- 独立部署。
- 独立版本控制。
- 自我管理。
- 自我恢复。

### 10.5 粗粒度

SOA 服务通常推荐粗粒度，尤其是暴露给企业外部或跨系统调用的服务。

原因：

```text
服务过细会导致调用频繁、网络开销大、耦合增加、治理复杂。
```

教材中特别提到：

```text
暴露在系统外部的服务推荐使用粗粒度接口；
较细粒度接口通常用于企业系统架构内部。
```

### 10.6 松耦合

服务使用者看到的是服务接口。

对服务使用者不可见的是：

- 服务位置。
- 实现技术。
- 当前状态。
- 私有数据。

考试关键词：

```text
接口依赖
实现隐藏
位置透明
松耦合
私有数据不可见
```

### 10.7 可重用

服务应该可以被多个业务流程和多个系统复用。

例如：

```text
客户查询服务可以被订单系统、客服系统、营销系统复用。
```

### 10.8 互操作性、兼容和策略声明

服务要支持不同平台、语言和系统之间协作。

通过策略描述服务的期望、约束和行为。

考试关键词：

```text
互操作
策略
兼容
跨平台
跨语言
```

## 11. SOA 的主要作用

教材中的重点表述：

```text
SOA 的主要作用是打破信息孤岛，把应用和资源转换成服务；
并把这些服务变成标准服务，形成资源共享。
```

你可以把 SOA 的价值总结为：

- 打破信息孤岛。
- 促进系统集成。
- 提高服务复用。
- 降低系统耦合。
- 支持业务流程编排。
- 提高企业 IT 架构灵活性。
- 支持异构系统互操作。
- 保护和复用遗留系统能力。

## 12. SOA 设计模式

### 12.1 服务注册表模式

服务注册表支持服务合同、策略和元数据的发布与管理。

三个动作：

```text
服务注册：服务提供者发布服务能力。
服务位置：服务消费者查询符合要求的服务。
服务绑定：消费者根据服务合同调用服务。
```

记忆：

```text
注册表模式 = 发布、查找、绑定。
```

### 12.2 企业服务总线模式

ESB 提供标准软件底层架构，让各种服务接入统一平台，通过标准消息方式交互。

核心功能：

- 位置透明的消息路由和寻址。
- 服务注册和命名管理。
- 多种消息范型，如请求/响应、发布/订阅。
- 支持多种传输协议。
- 支持多种数据格式及转换。
- 提供日志和监控。

考试中看到：

```text
位置透明
路由
寻址
发布/订阅
协议转换
格式转换
日志监控
```

基本就是 ESB。

### 12.3 服务编排模式

把多个细分业务服务按业务流程组织起来，形成更大的组合服务。

常见技术：

```text
BPEL
BPEL4WS
流程服务
服务组合
业务流程自动化
```

## 13. SOA 和微服务的区别

SOA 和微服务都强调服务，但它们不是一回事。

| 对比项 | SOA | 微服务 |
|---|---|---|
| 主要目标 | 企业系统集成、复用、流程编排 | 快速迭代、独立部署、弹性扩展 |
| 服务粒度 | 通常较粗 | 通常更细 |
| 部署方式 | 可集中治理，常见 ESB | 去中心化，服务独立部署 |
| 通信方式 | SOAP、WSDL、ESB、消息等 | REST、gRPC、消息队列等 |
| 数据管理 | 更关注跨系统集成 | 更强调每个服务自治和私有数据 |
| 治理方式 | 偏集中式治理 | 偏分布式治理 |
| 典型场景 | 企业级异构系统整合 | 互联网业务快速迭代和扩展 |

教材中的高频结论：

```text
微服务相比 SOA 更加精细。
微服务更多以独立进程方式存在，互相之间影响更小。
微服务接口更通用，例如 HTTP RESTful。
微服务更倾向于分布式、去中心化部署。
```

容易错的点：

```text
不能简单说“微服务一定比 SOA 高级”。
二者解决的问题侧重点不同。
SOA 更偏企业集成；
微服务更偏应用内部架构拆分、独立部署和敏捷交付。
```

## 14. SOA 和云原生的区别

因为你上一个专题学的是云原生，这里顺手对比一下。

| 对比项 | SOA | 云原生 |
|---|---|---|
| 关注点 | 企业应用集成与服务复用 | 应用在云环境中的弹性、韧性、自动化和治理 |
| 典型问题 | 信息孤岛、异构系统、跨系统流程 | 快速发布、自动扩缩容、容器编排、可观测 |
| 典型技术 | Web Service、SOAP、WSDL、UDDI、ESB、BPEL | 容器、K8s、微服务、Service Mesh、Serverless、DevOps |
| 治理方式 | 服务注册、服务总线、集中治理较多 | 平台化、自动化、声明式、服务网格 |
| 历史场景 | 企业信息化集成 | 云计算和互联网规模应用 |

记忆：

```text
SOA 重点解决“多个企业系统怎么集成”；
云原生重点解决“应用怎么天然适合云上运行和治理”。
```

## 15. 构建 SOA 时应该注意的问题

### 15.1 先识别集成需求

教材提到原有系统架构中的集成需求包括：

- 应用程序集成。
- 终端用户界面集成。
- 流程集成。
- 已有系统信息集成。

### 15.2 控制服务粒度

外部服务建议粗粒度，内部服务可以相对细粒度。

错误做法：

```text
把每个小函数都暴露成服务。
```

后果：

- 调用频繁。
- 网络开销大。
- 服务数量膨胀。
- 治理复杂。
- 业务边界混乱。

### 15.3 设计无状态服务

服务不依赖上次请求上下文。

这样更容易：

- 扩展。
- 负载均衡。
- 故障切换。
- 重试。
- 复用。

### 15.4 处理遗留系统

SOA 常用于整合旧系统。

典型方式：

- 通过适配器封装旧系统接口。
- 通过 ESB 进行协议转换。
- 把旧系统能力包装成服务。
- 逐步替换或演进遗留系统。

### 15.5 做好服务治理

服务一多，就必须治理。

治理内容包括：

- 服务目录。
- 服务版本。
- 服务权限。
- 服务监控。
- 服务 SLA。
- 服务调用审计。
- 服务变更管理。
- 服务退役管理。

可以按下面思路理解。

### 16.1 选择 SOA 解决方案

教材强调三个方面：

```text
尽量选择能进行全局规划的方案。
充分考虑企业自身需求。
从平台、实施等技术方面考察。
```

### 16.2 业务流程分析

关注：

- 建立服务模型。
- 建立业务对象。
- 建立服务接口。
- 建立服务流程。

服务建模方法：

```text
自顶向下分解法
业务目标分析法
自底向上分析法
```

### 16.3 设计服务模型

识别哪些业务能力可以封装为服务。

例如：

```text
客户信息服务

```

## 17. 用你的游戏项目理解 SOA

你是手机游戏开发工程师，熟悉 Unity、Cocos、Python、Orleans 和 MySQL，可以这样理解。

### 17.1 游戏业务中的 SOA 场景

一个游戏公司可能有很多系统：

- 账号系统。
- 支付系统。
- 防沉迷系统。
- 用户中心。
- GM 工具。
- 活动配置平台。
- 运营数据平台。
- 客服系统。
- 邮件系统。
- 道具发放系统。
- BI 报表系统。

这些系统可能不是一个项目组写的，也不一定用同一种语言。

SOA 思路是：

```text
账号系统暴露账号查询服务；
支付系统暴露支付确认服务；
防沉迷系统暴露实名校验服务；
邮件系统暴露发邮件服务；
道具系统暴露发奖服务；
运营平台通过服务编排完成活动奖励发放流程。
```

### 17.2 一个游戏活动发奖流程

例如运营要做“充值返利活动”。

SOA 风格可以拆成：

```text
查询玩家服务
查询充值服务
计算奖励服务
发放道具服务
发送邮件服务
记录运营日志服务
```

流程编排：

```text
玩家完成充值
-> 支付系统发布或提供支付确认服务
-> 活动系统调用奖励计算服务
-> 道具系统发放奖励
-> 邮件系统通知玩家
-> 数据平台记录活动效果
```

这就是“多个业务服务组合成一个业务流程”。

### 17.3 SOA 和 Orleans 的类比

Orleans 里的 Grain 是面向对象和虚拟 Actor 的抽象，适合构建分布式游戏服务。

SOA 里的服务是企业级业务能力抽象，重点是跨系统集成。

可以类比，但不要混同：

```text
Orleans Grain 更像应用内部的分布式对象或业务实体；
SOA 服务更像企业系统之间可复用、可注册、可调用的业务能力。
```

## 18. 选择题常见考点

### 18.1 SOA 定义

高频判断：

```text
SOA 是组件模型。
SOA 把不同功能单元称为服务。
服务通过定义良好的接口和契约联系。
接口应独立于硬件、操作系统和编程语言。
```

### 18.2 SOA 的主要作用

常考原话：

```text
打破信息孤岛，把应用和资源转换成服务；
把服务变成标准服务，形成资源共享。
```

### 18.3 Web Service 三件套

```text
SOAP：基于 XML 的消息交换协议。
WSDL：描述服务做什么、如何访问、位于何处。
UDDI：服务描述、发现和集成。
```

### 18.4 WSDL 三个属性

```text
服务做些什么。
如何访问服务。
服务位于何处。
```

看到选项中有“服务是否可用”，要警惕，它不是 WSDL 三个基本属性之一。

### 18.5 SOA 设计原则

必须背熟：

```text
无状态
单一实例
明确定义的接口
自包含和模块化
粗粒度
服务之间松耦合
重用能力
互操作性、兼容和策略声明
```

### 18.6 SOA 参考架构六类

```text
业务逻辑服务
控制服务
连接服务
业务创新和优化服务
开发服务
IT 服务管理
```

常见陷阱：

```text
中间件服务不是这六类之一。
```

### 18.7 ESB 功能

```text
消息路由
位置透明
服务注册和命名
请求/响应
发布/订阅
多协议支持
数据格式转换
日志和监控
```

### 18.8 SOA 与微服务

高频判断：

```text
微服务比 SOA 更细。
微服务常以独立进程存在。
微服务接口更常用 RESTful。
微服务更倾向于去中心化部署。
SOA 更偏企业集成和服务复用。
```

### 18.9 服务粒度

```text
外部暴露服务推荐粗粒度。
内部服务接口可以相对细粒度。
```

### 18.10 无状态服务

```text
服务请求应独立、自包含，不依赖前一次请求状态。
```

## 19. 案例题常见问法和答题模板

### 19.1 问：如何基于 SOA 改造企业系统

答题模板：

```text
1. 梳理现有系统和业务流程，识别应用集成、流程集成和信息集成需求。
2. 采用服务建模方法，按业务能力识别可复用服务，明确服务边界和服务粒度。
3. 定义服务契约，包括服务接口、输入输出、数据格式、调用协议、安全策略和 QoS。
4. 建立服务注册中心，实现服务发布、查找和绑定。
5. 通过 ESB 实现服务连接、消息路由、协议转换、数据格式转换和发布/订阅。
6. 通过 BPEL 或流程引擎对多个服务进行编排，形成端到端业务流程。
7. 对遗留系统通过适配器封装，将其能力包装为标准服务。
8. 建立服务治理体系，包括版本管理、权限控制、监控审计、SLA 管理和生命周期管理。
```

### 19.2 问：SOA 如何解决信息孤岛

答题模板：

```text
SOA 将各应用系统中的业务能力封装为标准服务，
通过统一接口和契约对外暴露。
服务消费者不需要了解服务内部实现，只需根据服务契约进行调用。
同时，借助服务注册中心和 ESB，可以实现服务发现、消息路由、协议转换和数据格式转换，
从而降低异构系统之间的耦合，促进服务复用和业务流程整合。
```

### 19.3 问：为什么 SOA 要使用 ESB

答题模板：

```text
ESB 为 SOA 提供统一的服务连接基础设施。
它可以实现位置透明的消息路由和寻址、服务注册和命名管理、请求/响应和发布/订阅等消息模式、
多协议支持、数据格式转换、日志监控和安全控制。
通过 ESB，服务消费者和服务提供者之间不需要直接点对点集成，
从而降低系统耦合，提高异构系统集成能力和可维护性。
```

### 19.4 问：如何控制 SOA 服务粒度

答题模板：

```text
SOA 服务粒度应结合业务边界、复用价值和调用频率确定。
对外暴露的服务应采用较粗粒度接口，封装完整业务能力，减少跨网络频繁调用。
企业内部可以使用相对细粒度服务支撑实现。
如果服务粒度过细，会导致服务数量膨胀、网络调用频繁、治理复杂和系统耦合增加。
```

### 19.5 问：SOA 与微服务如何比较

答题模板：

```text
SOA 和微服务都强调服务化和松耦合，但侧重点不同。
SOA 更关注企业级异构系统集成、服务复用和业务流程编排，常采用 ESB、SOAP、WSDL、UDDI 等技术。
微服务服务粒度更细，更强调围绕业务领域拆分、独立开发、独立部署和去中心化治理，
常采用 REST、gRPC、消息队列、容器和自动化运维等技术。
因此，SOA 偏企业集成，微服务偏互联网应用的敏捷交付和弹性扩展。
```

## 20. 论文可用表达

如果论文题目涉及 SOA、企业应用集成、系统集成、服务化改造，可以这样写。

```text
在项目中，原有系统存在接口不统一、系统耦合度高、业务流程跨系统协作困难等问题。
我采用面向服务架构思想，对现有系统的核心业务能力进行服务化封装，
将客户信息、订单处理、支付确认、库存查询等能力抽象为标准服务。
每个服务通过明确定义的接口和契约对外暴露，并注册到服务注册中心。
系统通过企业服务总线实现服务间的消息路由、协议转换、数据格式转换和日志监控，
避免系统之间形成点对点调用。
对于跨系统业务流程，通过流程引擎进行服务编排，实现端到端业务流程自动化。
同时建立服务治理机制，对服务版本、权限、安全、监控和 SLA 进行统一管理。
改造后，系统间耦合度降低，服务复用能力增强，新业务流程的上线周期缩短。
```
