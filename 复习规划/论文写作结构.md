# 系统架构设计师论文写作结构

## 1. 写作目标

论文不是写技术博客，也不是项目总结，而是证明自己具备系统架构设计能力。

考场论文要体现：

- 有真实或可信的项目背景。
- 能识别系统面临的主要问题。
- 能围绕题目选择合适的架构方案。
- 能说明关键技术措施。
- 能体现质量属性，如性能、可用性、安全性、可扩展性、可维护性。
- 能说明实施效果和反思。

最低目标：

- 不跑题。
- 结构完整。
- 项目可信。
- 技术点准确。
- 字数充足。

## 2. 统一项目背景

建议统一使用以下项目背景，后续所有论文围绕它改写。

项目名称：

> 面向移动游戏的高并发在线服务平台

项目描述：

> 我参与设计并建设了一套面向移动游戏的高并发在线服务平台，服务于多款 Unity/Cocos 开发的手机游戏。平台主要包括账号登录、角色管理、匹配组队、排行榜、商城支付、活动配置、邮件公告、数据统计和运营后台等模块。系统服务端采用分布式架构，核心业务服务基于 Orleans Actor 模型实现，数据层使用 MySQL 存储用户、角色、订单和道具等核心数据，Python 用于运营工具、数据处理和自动化脚本。系统上线后需要支撑大量玩家同时在线、频繁请求、弱网重连、活动峰值流量和版本快速迭代，因此对性能、可用性、扩展性、安全性和可维护性提出了较高要求。

## 3. 通用论文结构

建议按以下结构写所有论文：

1. 摘要
2. 项目背景
3. 问题与目标
4. 架构方案与关键措施
5. 实施效果
6. 总结反思

## 4. 字数与时间分配

论文考试时间紧，建议按以下节奏控制：

| 部分 | 建议字数 | 建议时间 |
|---|---:|---:|
| 审题和列提纲 | 不计入正文 | 10 分钟 |
| 摘要 | 250-350 字 | 15 分钟 |
| 项目背景 | 300-400 字 | 15 分钟 |
| 问题与目标 | 400-500 字 | 20 分钟 |
| 架构方案与关键措施 | 1200-1500 字 | 60 分钟 |
| 实施效果 | 300-400 字 | 15 分钟 |
| 总结反思 | 200-300 字 | 10 分钟 |
| 检查修改 | 不计入正文 | 5 分钟 |

总字数建议：

- 2500 字左右。
- 不低于 2200 字。
- 不盲目追求 3000 字以上，优先保证结构和主题贴合。

## 5. 摘要结构

摘要建议写成 4 句话：

1. 项目背景：我参与了什么项目。
2. 核心问题：项目面临什么架构挑战。
3. 解决方案：我采用了什么架构或技术措施。
4. 实施效果：取得了什么效果。

摘要模板：

```text
我参与设计并建设了一套面向移动游戏的高并发在线服务平台，该平台服务于多款 Unity/Cocos 开发的手机游戏，主要包括账号登录、角色管理、商城支付、活动配置、排行榜、数据统计和运营后台等模块。随着用户规模扩大和运营活动增多，系统面临高并发访问、活动峰值流量、业务快速迭代、服务故障隔离和数据一致性等问题。针对上述问题，我们围绕【论文主题】进行了架构设计，采用了【关键架构方案一】、【关键架构方案二】和【关键架构方案三】等措施，提升了系统的【质量属性一】、【质量属性二】和【质量属性三】。系统上线后，在多次版本发布和运营活动中保持了稳定运行，核心业务模块扩展性和可维护性得到明显提升。
```

替换说明：

- 【论文主题】：如架构风格、高可用、性能优化、安全架构、数据架构、微服务。
- 【关键架构方案】：如分层架构、微服务拆分、缓存、限流降级、读写分离、接口签名。
- 【质量属性】：如性能、可用性、安全性、可扩展性、可维护性。

## 6. 项目背景写法

项目背景要回答：

- 系统是什么。
- 给谁用。
- 有哪些核心功能。
- 技术环境是什么。
- 为什么有架构难度。

可复用段落：

```text
该平台主要面向移动游戏业务，客户端由 Unity 和 Cocos 开发，服务端负责玩家账号、角色、道具、订单、活动、排行榜等核心状态和业务规则。平台需要支持多款游戏接入，因此在设计时既要保证公共能力复用，又要允许不同游戏在活动、数值、运营策略上进行差异化扩展。系统采用分布式服务架构，核心业务服务使用 Orleans Actor 模型封装玩家、房间、活动等有状态对象，数据库采用 MySQL 存储核心业务数据，同时使用 Python 构建运营工具、自动化脚本和数据处理任务。
```

## 7. 问题与目标写法

这一部分要把题目引出来。不要泛泛说“系统复杂”，要结合论文主题写具体问题。

通用问题库：

- 玩家同时在线人数增长，登录、角色、活动、排行榜请求量明显上升。
- 大型运营活动开始时存在瞬时峰值，容易造成服务压力集中。
- 手游弱网和重连场景多，要求服务端具备幂等和状态恢复能力。
- 商城支付、道具发放等链路要求较强的数据一致性和可追溯性。
- 游戏版本迭代频繁，活动规则变化快，需要架构支持快速发布和低风险变更。
- 多个业务模块如果耦合过高，会导致故障扩散和团队协作效率下降。
- 账号、支付、运营后台等模块涉及安全风险，需要统一认证、授权和审计。

目标表达：

```text
因此，本项目的架构设计目标不是单纯实现业务功能，而是在满足功能需求的基础上，重点提升系统的性能、可用性、可扩展性、安全性和可维护性，并在一致性、复杂度和开发效率之间进行合理权衡。
```

## 8. 架构方案与关键措施写法

主体部分通常写 3-4 个措施，每个措施都按以下顺序展开：

```text
措施名称
为什么需要这个措施
具体怎么做
解决了什么问题
带来了什么效果或取舍
```

例如：

```text
第一，采用分层架构明确系统职责边界。我们将系统划分为接入层、业务服务层、数据访问层和基础设施层。接入层负责协议适配、鉴权、限流和请求转发；业务服务层承载账号、角色、商城、活动、排行榜等核心业务逻辑；数据访问层封装 MySQL 访问和事务处理；基础设施层提供日志、监控、配置、任务调度等公共能力。通过分层设计，系统职责更加清晰，降低了客户端协议变化、业务规则变化和数据访问实现之间的相互影响。
```

## 9. 六类主题的主体结构

### 9.1 架构风格

可写 4 个措施：

1. C/S 架构：客户端负责表现和交互，服务端负责核心状态和规则。
2. 分层架构：接入层、业务服务层、数据访问层、基础设施层。
3. 微服务架构：账号、角色、商城、活动、排行榜、运营后台服务拆分。
4. Actor 模型：封装玩家、房间、匹配等有状态对象，降低并发冲突。

重点质量属性：

- 可维护性。
- 可扩展性。
- 性能。
- 故障隔离。

### 9.2 高可用架构

可写 4 个措施：

1. 服务集群部署，避免单点故障。
2. 登录、支付、活动等核心服务故障隔离。
3. 限流、降级、熔断保护核心链路。
4. MySQL 备份恢复和关键数据审计。

重点质量属性：

- 可用性。
- 可靠性。
- 容错性。
- 可恢复性。

### 9.3 性能优化

可写 4 个措施：

1. 热点数据缓存，如配置、活动、排行榜。
2. 异步处理日志、邮件、活动结算等非实时任务。
3. MySQL 索引优化和 SQL 优化。
4. 活动峰值前缓存预热和容量评估。

重点质量属性：

- 响应时间。
- 吞吐量。
- 并发能力。
- 资源利用率。

### 9.4 安全架构

可写 4 个措施：

1. 账号认证和 Token 管理。
2. 接口签名、时间戳和随机数防重放。
3. 支付回调校验和道具发放幂等。
4. 后台权限控制和操作审计。

重点质量属性：

- 机密性。
- 完整性。
- 可用性。
- 可审计性。

### 9.5 数据架构

可写 4 个措施：

1. 核心业务数据建模，如账号、角色、订单、道具。
2. MySQL 事务保证支付和道具发放一致性。
3. 读写分离或冷热数据分离。
4. 日志和统计数据异步采集，避免影响核心链路。

重点质量属性：

- 一致性。
- 可扩展性。
- 可维护性。
- 查询性能。

### 9.6 微服务架构

可写 4 个措施：

1. 按业务能力拆分服务。
2. 服务接口和协议治理。
3. 服务独立部署和横向扩展。
4. 服务监控、限流和故障隔离。

重点质量属性：

- 可扩展性。
- 可维护性。
- 独立部署能力。
- 故障隔离能力。

## 10. 实施效果写法

实施效果要具体，但不用写夸张数字。可以写相对效果。

可复用表达：

```text
通过上述设计，平台在后续多个游戏版本发布和运营活动中保持了较好的稳定性。账号、角色、商城、活动和排行榜等模块边界更加清晰，新活动上线时主要修改活动服务和配置数据，对其他核心服务影响较小。针对活动峰值流量，系统可以通过增加服务实例、提前预热缓存和配置限流策略进行应对。对于支付和道具发放等关键链路，通过事务控制、幂等处理和日志审计，降低了重复发放和数据不一致的风险。总体来看，系统的性能、可用性、扩展性和可维护性均得到提升。
```

## 11. 总结反思写法

总结反思要体现架构权衡，不能只说“效果很好”。

可复用表达：

```text
当然，该架构设计也带来了一定复杂度。例如，服务拆分后需要投入更多精力进行接口治理、监控告警和问题定位；缓存和异步处理提升了性能，但也要求我们更加重视数据一致性和异常补偿机制。因此，在架构设计过程中不能孤立追求某一项技术先进性，而应结合业务规模、团队能力和质量属性要求，在性能、可用性、一致性、复杂度和成本之间进行权衡。通过本项目实践，我认识到系统架构设计的核心是围绕业务目标和质量属性做出可落地的技术取舍。
```

## 12. 考场检查清单

写完后用 2-3 分钟检查：

- [ ] 题目关键词是否在摘要和正文多次出现。
- [ ] 是否写清楚项目背景。
- [ ] 是否至少写了 3 个关键措施。
- [ ] 每个措施是否有“为什么、怎么做、效果”。
- [ ] 是否体现质量属性。
- [ ] 是否有实施效果。
- [ ] 是否有架构权衡和反思。
- [ ] 是否避免写成纯客户端开发经验。

## 14. 常用连接句

用于让文章更像论文，而不是项目流水账：

```text
针对上述问题，我们从架构层面对系统进行了重新设计。
```

```text
该设计的核心目的不是单纯拆分功能，而是提升系统在高并发和频繁迭代场景下的可扩展性和可维护性。
```

```text
在该方案中，我们重点考虑了性能、可用性和数据一致性之间的平衡。
```

```text
实践表明，该方案能够较好地支撑业务增长，但也带来了监控、治理和问题定位方面的复杂度。
```

```text
因此，架构设计需要结合业务特点、团队能力和运行环境进行综合权衡。
```
