## 正文

### 一、项目背景 <span style="font-size: 0.75em; color: #666;">（329 字）</span>

我所在团队负责建设一套移动游戏高并发在线服务平台，为多款手机游戏提供统一的服务端能力。客户端由 Unity 和 Cocos 开发，服务端采用 Orleans 分布式框架，MySQL 存储账号、角色、订单、道具和活动记录等核心数据。平台主要业务包括登录、角色加载、活动配置、活动领奖、商城支付、排行榜、邮件和日志统计。

移动游戏业务的性能压力集中在几个场景。开服和版本更新时，大量玩家同时登录并加载角色；限时活动开始时，玩家集中拉取活动配置、领取奖励和查询排行榜；商城促销时，订单和支付请求明显增加。早期系统在普通流量下运行正常，但活动高峰时会出现接口响应变慢、数据库慢 SQL 增多、消息队列积压和部分服务实例负载过高等问题。因此，我们将性能优化作为平台改造重点。我主要负责核心链路压测、数据库和缓存优化方案、异步化改造以及性能监控指标设计。

### 二、性能优化目标和分析方法 <span style="font-size: 0.75em; color: #666;">（299 字）</span>

性能优化首先要明确目标。我们将目标分为用户体验指标和系统资源指标两类。用户体验指标包括接口平均响应时间、95% 响应时间、错误率和超时率；系统资源指标包括 QPS、TPS、CPU、内存、数据库连接数、慢 SQL、缓存命中率和消息队列积压量。

在分析方法上，我们不直接凭经验优化，而是先梳理核心业务链路。例如玩家进入活动页时，需要登录态校验、角色信息读取、活动配置查询、资格判断和奖励状态查询。我们通过压测和链路追踪观察每一步耗时，判断瓶颈在业务服务、数据库、缓存还是外部依赖。

性能优化还需要区分核心和非核心链路。登录、支付、道具发放和活动领奖属于核心链路，必须优先保证响应和正确性；日志、运营统计、邮件通知和部分展示刷新可以异步或延迟处理。明确优先级后，优化措施才能有重点。

### 三、性能优化的关键设计措施 <span style="font-size: 0.75em; color: #666;">（614 字）</span>

第一，使用缓存降低热点读取压力。活动配置、商城商品、排行榜展示、公告和部分角色展示数据属于读多写少场景，我们将其放入缓存，并在大型活动前进行预热。对于活动配置，采用版本号控制更新；对于排行榜，采用分区缓存和定时刷新，避免每次查询都访问数据库。

第二，进行数据库 SQL 和索引优化。订单、活动记录和排行榜相关表随着数据增长，部分查询出现慢 SQL。我们通过慢 SQL 日志和执行计划分析，补充组合索引，避免全表扫描；对运营后台的大分页查询改为按时间和主键分页；对统计类查询改为异步聚合，避免直接压核心库。

第三，将非核心任务异步化。早期部分接口同步写日志、发送邮件和更新统计，导致响应时间增加。我们将日志、邮件、运营统计和部分活动结算任务写入消息队列，由消费者异步处理。核心接口只完成必要状态变更和任务投递，从而缩短用户请求链路。

第四，通过服务扩容和负载均衡提升吞吐量。账号、活动、排行榜和商城服务以多实例部署，网关根据健康状态和权重分发流量。活动高峰前，根据压测结果提前扩容活动服务、缓存和队列消费者。对于玩家状态和活动对象，利用 Orleans Actor 模型按用户或活动分区，减少状态冲突。

第五，采用限流和降级保护系统。活动领奖、排行榜刷新和运营查询等接口在高峰时可能流量过大。我们按用户、服务器、接口设置限流，对非核心排行榜实时刷新、详细统计展示等功能进行降级，优先保障登录、支付和活动领奖。

第六，建立性能测试和监控闭环。每次大型活动前，我们用接近真实的数据规模进行压测，观察响应时间、数据库压力和队列积压。上线后持续监控核心指标，发现瓶颈后再进行回归压测验证优化效果。

### 四、实施过程中的问题和解决措施 <span style="font-size: 0.75em; color: #666;">（481 字）</span>

第一个问题是缓存一致性。活动配置和排行榜数据进入缓存后，可能出现数据库已更新但缓存未刷新。我们通过版本号、过期时间和主动刷新机制解决，对强一致要求高的支付和资产数据则不使用缓存作为最终依据。

第二个问题是异步化带来的最终一致。邮件、日志和统计可以延迟，但道具发放和活动领奖需要保证最终正确。我们通过任务状态、幂等键和补偿机制处理消费失败和重复消费。

第三个问题是局部热点。某些热门活动或排行榜会集中到少数服务实例。我们按服务器和活动编号进行分区，并对热点榜单做缓存拆分，降低局部压力。

第四个问题是优化效果验证。单独优化某个 SQL 或增加实例不一定能改善整体性能。我们坚持通过压测和监控数据验证优化效果，避免凭感觉判断。

第五个问题是性能和一致性之间的取舍。排行榜、活动展示和公告可以接受短暂延迟，因此适合缓存和异步刷新；但支付订单、道具资产和领奖状态必须以数据库最终结果为准。我们根据业务重要性划分强一致和最终一致场景，避免为了追求性能而破坏核心数据正确性。

第六个问题是客户端弱网导致的重复请求。部分玩家在网络不稳定时会重复点击领奖或购买按钮。我们在客户端增加请求状态提示，在服务端增加幂等键和重复请求保护，既减少无效请求，又避免重复发放。

### 五、实施效果 <span style="font-size: 0.75em; color: #666;">（408 字）</span>

经过性能优化后，平台在活动高峰期的响应稳定性明显提升。活动配置和排行榜查询命中缓存后，数据库读压力下降；SQL 和索引优化减少了慢查询；日志、邮件和统计异步化后，核心接口链路缩短；服务扩容和负载均衡提升了并发处理能力；限流降级避免了异常流量拖垮核心服务。

在一次限时活动中，玩家集中进入活动页并查询排行榜。由于提前进行了缓存预热和队列消费者扩容，活动服务和数据库都保持在可控范围内，接口超时率明显低于改造前。压测数据也为后续活动容量规划提供了依据。

性能优化后，团队也形成了较固定的活动上线流程：活动前根据预约人数和历史数据进行容量评估，提前预热缓存并扩容关键服务；活动中观察接口耗时、错误率、缓存命中率和队列积压；活动后复盘慢 SQL、热点接口和容量使用情况。通过这种闭环，性能优化不再是出现问题后的临时处理，而成为版本发布和活动运营的一部分。

同时，我们认识到性能优化是持续过程。随着数据量增长、活动规则变化和用户规模扩大，原有瓶颈可能再次出现。因此，性能测试、监控分析和容量评估必须成为常态。

### 六、总结 <span style="font-size: 0.75em; color: #666;">（189 字）</span>

系统性能优化设计是保障高并发业务稳定运行的重要工作。在本项目中，我们围绕移动游戏在线服务平台，从缓存、数据库、异步处理、服务扩容、负载均衡、限流降级、性能测试和监控等方面进行优化，解决了活动高峰响应慢、数据库压力大和非核心任务阻塞核心链路等问题。实践表明，性能优化不能只依赖增加硬件资源，而应基于业务链路和监控数据定位瓶颈，综合运用缓存、索引、异步化和架构扩展等手段。架构师需要在性能、一致性、复杂度和成本之间进行权衡。
