一、项目背景

- 项目：移动游戏运营自动化与活动任务平台。
- 业务：活动配置发布、支付回调、礼包核销、邮件发放、日志清洗、运营报表。
- 技术：Unity/Cocos 客户端、Orleans 服务端、MySQL、消息队列、对象存储、云函数。
- 问题：任务峰谷明显，低频任务常驻部署浪费资源，活动高峰又容易积压。

二、Serverless 核心组成

- FaaS：函数计算，负责短时、事件触发的业务逻辑。
- BaaS：云数据库、对象存储、消息队列、日志服务、配置服务等后端能力。
- 触发器：HTTP、定时器、消息队列、对象存储事件、数据库事件。
- 治理能力：权限、监控、日志、告警、灰度和补偿。

三、设计方案

- API 网关统一接入支付回调和运营后台请求。
- 云函数处理支付校验、礼包核销、活动结算、日志清洗。
- 消息队列削峰，承接邮件、统计、奖励补发等异步任务。
- 状态外置到 MySQL、缓存和对象存储。
- 通过监控告警和幂等补偿保障可靠性。

四、落地难点

- 冷启动影响响应。
- 函数无状态，需要外部化状态。
- 分布式链路调试困难。
- 重试可能导致重复发放。
- 按量计费需要关注执行时间和调用量。

五、实施效果

- 闲时资源成本降低。
- 活动高峰自动扩展。
- 后台任务与核心游戏服务解耦。
- 运维工作量下降。

## 摘要 <span style="font-size: 0.75em; color: #666;">（379 字）</span>

我参与设计并改造了一套面向移动游戏的运营自动化与活动任务平台，该平台服务于多款 Unity/Cocos 开发的手机游戏，主要承担活动配置发布、支付回调校验、礼包码核销、邮件发放、日志清洗、运营报表和定时结算等任务。项目早期采用常驻服务和定时脚本实现这些能力，存在低频任务资源浪费、活动高峰处理积压、部署维护成本高、任务失败难以追踪等问题。针对这些问题，我们引入 Serverless 架构模式，将短时、离散、事件触发明显的后台任务改造为云函数，并结合 API 网关、消息队列、对象存储、云数据库、日志服务和监控告警等 BaaS 能力，实现按需执行、弹性伸缩和运维简化。本文结合该项目，论述 Serverless 架构的核心组成、各组件作用、如何协同实现降本增效和弹性伸缩，以及方案选型依据、落地难点和实施效果。实践表明，Serverless 适合运营自动化、异步任务和峰谷明显的业务场景，但需要重点处理冷启动、状态外置、幂等补偿、权限控制和可观测性等问题。

## 正文

### 一、项目背景 <span style="font-size: 0.75em; color: #666;">（328 字）</span>

我所在团队负责建设一套移动游戏运营自动化与活动任务平台，为多款手机游戏提供运营后台能力。客户端主要由 Unity 和 Cocos 开发，核心游戏服务基于 Orleans 分布式框架实现，MySQL 存储账号、角色、订单、道具和活动记录等核心数据。运营平台则负责活动配置发布、礼包码核销、支付回调校验、奖励结算、邮件发放、日志清洗、数据统计和运营报表生成。

在平台早期，这些任务主要通过常驻服务和定时脚本实现。随着游戏数量增加和运营活动增多，问题逐渐显现：一些任务平时调用很少，却需要长期部署服务实例；活动开始或结束时，礼包核销、活动结算、邮件发放和日志处理又会在短时间内集中爆发；多个脚本分散部署，失败后缺少统一日志和补偿机制；运维人员需要维护大量小服务，成本较高。基于这些特点，我们决定将部分后台任务改造为 Serverless 架构。

### 二、Serverless 架构的核心组成 <span style="font-size: 0.75em; color: #666;">（390 字）</span>

Serverless 并不是没有服务器，而是由云平台屏蔽服务器采购、部署、扩容和运行维护等底层细节，开发团队主要关注业务函数、事件触发和后端服务组合。在本项目中，Serverless 架构主要由 FaaS、BaaS、事件触发器、API 网关、监控告警和权限控制组成。

FaaS 即函数即服务，是业务逻辑的执行单元。我们将支付回调校验、礼包码核销、活动结算、邮件任务生成、日志清洗和报表计算拆分为多个独立函数。函数只在事件触发时运行，执行结束后释放资源。

BaaS 即后端即服务，提供函数依赖的基础能力，包括云数据库、对象存储、消息队列、日志服务、配置中心和鉴权服务。活动配置和日志文件存放在对象存储中，任务状态、幂等记录和补偿记录写入 MySQL，异步任务通过消息队列传递。

事件触发器负责启动函数执行。运营后台请求通过 API 网关触发函数，支付平台回调通过 HTTP 触发函数，活动结算通过定时器触发函数，日志文件上传对象存储后触发清洗函数，邮件和奖励发放任务则通过消息队列触发。

### 三、架构方案设计与实现 <span style="font-size: 0.75em; color: #666;">（577 字）</span>

第一，使用 API 网关统一管理外部入口。支付平台回调、运营后台操作和内部工具请求都先进入 API 网关，由网关完成鉴权、限流、参数校验和路由转发。对于支付回调，我们在网关层进行来源校验和基础限流，再转发给支付校验函数，避免函数直接暴露在公网环境中。

第二，将短时业务逻辑拆分为云函数。以礼包核销为例，运营人员在后台创建礼包活动后，礼包码文件上传到对象存储并触发导入函数。玩家兑换礼包时，请求通过网关触发核销函数，函数校验活动状态、礼包码、用户资格和幂等记录，再写入核销结果。对于邮件发放和运营统计，函数不直接同步完成，而是投递消息到队列，由后续函数异步处理。

第三，利用消息队列削峰和解耦。活动结束后可能同时产生大量奖励结算和邮件发放任务。如果同步处理，会拖慢运营后台和核心游戏服务。我们将结算结果写入消息队列，由邮件函数、统计函数和补偿函数按各自能力消费。队列既能缓冲高峰流量，也能隔离不同任务之间的失败影响。

第四，所有状态外置。Serverless 函数实例可能随时创建和销毁，不能依赖本地内存保存状态。因此，订单状态、任务状态、函数执行结果、重试次数和补偿记录全部存放到数据库或缓存中。函数每次执行都从外部存储读取必要状态，并通过幂等键避免重复处理。

第五，建立监控、告警和补偿机制。我们为每类函数设置执行次数、平均耗时、错误率、超时次数、重试次数和队列积压等指标。对于支付、礼包和奖励发放等关键任务，函数必须记录处理状态。若多次执行失败，则进入补偿表，由后台工具触发人工或自动补偿。

### 四、选型依据和落地难点 <span style="font-size: 0.75em; color: #666;">（442 字）</span>

我们选择 Serverless 的依据主要有三点。第一，任务具有明显事件触发特征，如支付回调、对象存储上传、定时结算和消息消费。第二，任务执行时间较短，且多数不需要长期保持连接或本地状态。第三，业务流量峰谷明显，按峰值部署常驻服务会造成资源浪费。

落地过程中，第一个难点是冷启动。函数首次执行时需要初始化运行环境和依赖库，可能增加响应时间。对此，我们控制函数依赖规模，避免把多个无关任务写进一个大函数，并在大型活动前对关键函数进行预热。

第二个难点是状态一致性。函数重试可能导致礼包重复核销或奖励重复发放。我们通过用户编号、活动编号、订单编号、礼包码和事件编号组成幂等键，并利用数据库唯一约束和状态机控制处理流程。

第三个难点是链路追踪。Serverless 任务分布在网关、函数、消息队列和数据库之间，如果缺少统一请求 ID，问题排查会很困难。我们在入口处生成 traceId，并在日志、消息和数据库记录中传递。

第四个难点是成本控制。Serverless 按量计费，但如果函数执行时间长、日志量大或失败重试过多，也会增加成本。因此，我们对函数超时时间、并发上限、日志级别和重试次数进行限制。

### 五、实施效果 <span style="font-size: 0.75em; color: #666;">（222 字）</span>

改造后，运营自动化平台的弹性和运维效率明显提升。低频任务不再需要长期占用服务器资源，活动高峰期间云函数能够根据事件量自动扩展，邮件发放、日志清洗和活动结算等异步任务的处理能力更容易按需调整。核心游戏服务不再承载大量运营后台任务，故障隔离能力提高。

同时，我们也认识到 Serverless 不是万能方案。对于长连接网关、实时战斗、玩家状态管理等低延迟和强状态场景，仍然采用常驻服务更合适。Serverless 更适合作为整体架构中的弹性补充，用于短时、离散、事件驱动和运维成本敏感的任务。

### 六、总结 <span style="font-size: 0.75em; color: #666;">（237 字）</span>

Serverless 架构模式通过 FaaS 和 BaaS 的组合，使开发团队能够以事件驱动方式实现业务功能，并将服务器运维、弹性扩缩容和部分基础能力交给云平台。在本项目中，我们将其用于移动游戏运营自动化平台中的支付回调、礼包核销、活动结算、邮件发放和日志清洗等场景，实现了降本增效和弹性伸缩。实践表明，Serverless 的关键价值在于按需运行和降低运维负担，但架构设计时必须处理冷启动、状态外置、幂等补偿、可观测性和权限控制等问题。合理的做法不是全面替换传统服务，而是根据业务特点选择适合 Serverless 的边界。
