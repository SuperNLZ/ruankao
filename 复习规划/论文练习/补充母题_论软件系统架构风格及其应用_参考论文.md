## 题目

论软件系统架构风格及其应用 <span style="font-size: 0.75em; color: #666;">（全文约 2592 字，提纲不计）</span>

## 考场提纲 <span style="font-size: 0.75em; color: #666;">（330 字）</span>

说明：提纲用于考前构思和考场草稿，不计入论文正文。

一、项目背景

- 项目：移动游戏客户端与在线服务协同平台。
- 模块：客户端 UI、资源加载、网络通信、登录、角色、活动、商城、排行榜、运营后台。
- 技术：Unity/Cocos 客户端、Orleans 服务端、MySQL、缓存、消息队列。
- 问题：客户端和服务端职责不清、模块耦合高、活动迭代快、服务扩展困难。

二、架构风格选择

- C/S 架构：明确客户端与服务端职责。
- 分层架构：接入层、业务层、数据层、基础设施层。
- MVC 架构：客户端 UI、数据和控制逻辑分离。
- 事件驱动架构：模块之间通过事件解耦。
- 模块化架构：公共能力封装和程序集隔离。

三、应用过程

- 梳理业务边界。
- 划分客户端和服务端职责。
- 重构客户端表现层、逻辑层和数据层。
- 拆分服务端账号、活动、商城、排行榜等模块。
- 通过事件和接口降低依赖。

四、效果

- 模块边界清晰。
- 迭代效率提升。
- 代码耦合降低。
- 服务扩展和维护更容易。

## 摘要 <span style="font-size: 0.75em; color: #666;">（359 字）</span>

我参与设计并重构了一套移动游戏客户端与在线服务协同平台，该平台服务于多款 Unity/Cocos 开发的手机游戏，主要包括客户端 UI、资源加载、网络通信、登录、角色、活动、商城、排行榜和运营后台等模块。项目早期由于缺少统一架构规划，存在客户端表现逻辑和业务逻辑耦合、网络请求分散、服务端模块边界不清、活动迭代影响范围大、公共能力重复实现等问题。针对这些问题，我们围绕软件系统架构风格进行分析和设计，结合项目特点采用 C/S 架构、分层架构、MVC 架构、事件驱动架构和模块化架构：通过 C/S 架构明确客户端和服务端职责，通过分层架构规范服务端结构，通过 MVC 和事件机制降低客户端模块耦合，通过模块化封装提升公共能力复用。本文结合该项目，论述架构风格的选择依据、具体应用过程和实施效果。实践表明，合理组合架构风格能够提升系统可维护性、可扩展性和开发效率，但架构风格选择必须服务于业务目标和团队能力。

## 正文

### 一、项目背景 <span style="font-size: 0.75em; color: #666;">（296 字）</span>

我所在团队负责建设一套移动游戏客户端与在线服务协同平台。客户端主要采用 Unity 和 Cocos 开发，服务端采用 Orleans 分布式框架，MySQL 存储账号、角色、订单、道具和活动数据。平台支撑登录、角色、战斗、活动、商城支付、排行榜、邮件、资源热更和运营后台等功能。

在项目早期，各模块主要按功能快速堆叠。客户端 UI 代码中混杂了网络请求、业务判断和数据更新逻辑；服务端账号、活动、商城、排行榜等能力边界不清；公共组件被多个模块重复实现；活动规则变化时，经常需要同时修改 UI、逻辑、数据和服务端接口。随着版本迭代加快，系统可维护性和扩展性逐渐下降。因此，我们决定从架构风格层面对系统进行梳理和改造。我在项目中主要负责客户端分层重构、网络模块封装和服务端业务边界设计。

### 二、架构风格选择依据 <span style="font-size: 0.75em; color: #666;">（430 字）</span>

架构风格是对系统组织方式和组件交互方式的抽象。选择架构风格时，不能只看技术流行程度，而要结合系统特点、质量属性和团队能力。

首先，移动游戏天然适合 C/S 架构。客户端负责界面展示、动画表现、玩家交互和部分本地缓存，服务端负责账号认证、资产数据、支付校验、活动规则和排行榜计算等核心业务。这样可以避免关键数据和规则完全暴露在客户端，提升安全性和一致性。

其次，服务端需要采用分层架构。平台功能较多，如果业务逻辑、数据访问、基础组件混杂在一起，会导致维护困难。分层架构能够明确接入层、业务服务层、数据访问层和基础设施层的职责。

第三，客户端 UI 适合采用 MVC 或类似分离思想。游戏 UI 多、变化快，如果表现层直接操作数据和业务逻辑，后续修改成本高。通过模型、视图和控制逻辑分离，可以降低 UI 修改对业务逻辑的影响。

第四，活动、邮件、排行榜和日志等模块适合采用事件驱动思想。某个业务动作发生后，其他模块通过事件订阅处理后续任务，可以减少直接调用。

第五，客户端和服务端都需要模块化。公共网络、资源加载、配置解析、日志、支付和 SDK 接入等能力应封装为独立模块，降低重复代码和编译成本。

### 三、架构风格的具体应用 <span style="font-size: 0.75em; color: #666;">（528 字）</span>

第一，应用 C/S 架构明确职责边界。客户端主要负责交互、表现、资源加载和本地状态展示；服务端负责账号、角色、订单、道具、活动资格和排行榜等核心状态。对于支付、活动领奖和道具发放等关键操作，客户端只发起请求和展示结果，最终校验和状态变更均由服务端完成。

第二，服务端采用分层架构。接入层负责协议适配、鉴权、限流和请求转发；业务服务层负责账号、角色、商城、活动、排行榜等业务逻辑；数据访问层封装 MySQL 操作、事务和缓存访问；基础设施层提供日志、配置、监控、消息队列和任务调度能力。分层后，各层职责清晰，接口变化和数据访问变化不再直接影响所有模块。

第三，客户端采用 MVC 思想改造 UI。我们将界面显示逻辑放在 View，将角色信息、活动进度和商城商品等数据放在 Model，将按钮点击、网络请求和状态流转放在 Controller 或 Presenter 中。UI 只订阅数据变化并刷新表现，避免直接修改核心业务数据。

第四，采用事件驱动降低模块依赖。例如玩家完成支付后，商城模块发布支付成功事件，背包、邮件、日志和统计模块分别订阅处理。客户端内部也通过事件通知 UI 刷新、红点变化和资源加载状态，减少模块之间直接引用。

第五，采用模块化架构管理公共能力。客户端使用程序集和接口封装网络、资源、配置、SDK 接入和日志模块，服务端也将公共组件沉淀为基础库。模块化后，编译速度和复用能力提升，依赖关系也更容易检查。

### 四、实施过程中的问题和解决措施 <span style="font-size: 0.75em; color: #666;">（362 字）</span>

第一个问题是架构改造不能影响正在进行的版本交付。我们采用渐进式重构，先从网络模块、配置模块和活动 UI 等边界清晰的部分开始，再逐步改造核心模块。

第二个问题是旧代码依赖复杂。部分 UI 直接访问业务数据和网络接口，短期内无法完全拆开。我们通过适配层过渡，新代码使用新架构，旧代码逐步迁移。

第三个问题是事件驱动可能造成调用链不直观。为避免事件滥用，我们规定只有跨模块通知和异步任务使用事件，模块内部流程仍保持清晰的函数调用，并为关键事件增加日志。

第四个问题是服务端分层后接口数量增加。我们通过统一接口规范、错误码和文档管理降低协作成本。

第五个问题是客户端和服务端协作方式需要调整。过去客户端经常根据界面需要临时要求服务端增加字段，导致接口膨胀。改造后，我们先从业务对象和状态流转出发设计接口契约，再由客户端根据契约进行展示适配。这样虽然前期沟通成本略有增加，但接口稳定性和后续维护性更好。

### 五、实施效果 <span style="font-size: 0.75em; color: #666;">（387 字）</span>

通过架构风格改造，系统结构更加清晰。客户端 UI、数据和控制逻辑分离后，界面修改对业务逻辑的影响明显减少。网络、资源和配置等公共模块封装后，重复代码减少，编译和定位问题更方便。服务端分层后，账号、商城、活动和排行榜等模块边界更清楚，新活动上线时主要修改活动服务和配置，对其他模块影响较小。事件驱动机制也让邮件、日志和统计等非核心任务与核心链路解耦。

在后续版本迭代中，新的限时活动主要通过活动配置、服务端活动逻辑和少量客户端展示组件完成，网络、资源和账号等基础模块基本不需要修改。客户端 UI 调整时，也可以通过事件和数据绑定刷新界面，减少对业务对象的直接操作。整体来看，系统从“功能堆叠”逐步转向“按职责组织”，团队协作效率和问题定位效率都有提升。

同时，我们认识到架构风格不是越多越好。每种风格都有适用边界，事件驱动过度使用会增加调试难度，分层过多会增加代码跳转成本，模块化过细会增加依赖管理成本。因此，架构设计需要结合业务规模、团队能力和质量属性要求进行取舍。

### 六、总结 <span style="font-size: 0.75em; color: #666;">（184 字）</span>

软件系统架构风格是指导系统组织和演进的重要手段。在本项目中，我们围绕移动游戏客户端与在线服务协同平台，综合采用 C/S 架构、分层架构、MVC 架构、事件驱动架构和模块化架构，解决了职责不清、耦合过高、重复实现和扩展困难等问题。实践表明，合理的架构风格组合能够提升系统可维护性、可扩展性和开发效率。但架构风格不能脱离业务场景孤立选择，架构师应围绕项目目标、团队能力和质量属性要求，在复杂度和收益之间做出平衡。
