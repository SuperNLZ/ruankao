# 《论软件系统架构风格及其应用》修改建议

## 1. 当前评价

当前版本已经有了提纲和摘要，说明论文已经启动，这是正确的第一步。

但目前整体更像“Unity 客户端代码重构经验”，还不够像“系统架构设计师论文”。本题关键词是“软件系统架构风格及其应用”，所以文章重点应放在架构风格的选择、组合使用和实施效果，而不是只写 asmdef、MVC、事件分发等具体实现手段。

## 2. 主要问题

### 2.1 项目层级偏低

当前写法：

> 我参与并设计一款基于 Unity 引擎的 3D 手机游戏的开发。

这个表达容易让阅卷人认为文章只是在写客户端开发或代码整理，架构层级不够。

建议提升为：

> 我参与设计并重构了一款基于 Unity 引擎的 3D 移动游戏客户端及其在线服务接入架构。

这样既保留你的真实 Unity 背景，又能自然写到：

- C/S 架构
- 分层架构
- MVC 架构
- 事件驱动架构
- 模块化架构
- 客户端与服务端职责划分

### 2.2 架构风格不够明确

当前提到了 MVC、事件驱动和 asmdef，但没有明确点出“架构风格”。

建议正文中明确写出以下 5 类架构风格：

- C/S 架构：客户端负责表现和交互，服务端负责核心数据和业务规则。
- 分层架构：表现层、业务逻辑层、数据层、基础设施层。
- MVC 架构：视图、模型、控制逻辑分离。
- 事件驱动架构：通过事件分发降低模块之间直接依赖。
- 模块化架构：通过 asmdef、接口和公共组件封装降低编译和维护成本。

### 2.3 摘要语言不够正式

当前摘要中有一些表达不适合考试论文：

- “参与设计并设计”重复。
- “遇到了不少的问题”偏口语。
- “模块之前”应改为“模块之间”。
- “项目结构清爽了很多”不够论文化。

建议使用更正式的表达：

- “存在目录结构混乱、模块职责不清、表现层与业务逻辑耦合严重等问题”
- “导致功能迭代效率下降，缺陷定位和维护成本较高”
- “系统模块边界更加清晰，代码耦合度降低，编译和迭代效率提升”

### 2.4 正文存在模板残留

当前文件中 `## 正文` 后面的标题包括：

- 负载均衡的设计目标
- 静态、动态和基于场景的负载均衡
- 负载均衡方案设计与实现

这些明显是之前负载均衡范文的残留内容，和“架构风格”题目不匹配。

建议在正式写正文前删除这些残留小节，重新按“架构风格”主题组织正文。

## 3. 建议修改后的摘要

```md
## 摘要 <span style="font-size: 0.75em; color: #666;">（约 300 字）</span>

我参与设计并重构了一款基于 Unity 引擎的 3D 移动游戏客户端及其在线服务接入架构。该项目包含角色系统、战斗系统、UI 系统、资源加载、网络通信、活动配置和数据同步等模块。项目早期由于缺少统一的架构规划，存在目录结构混乱、模块职责不清、表现层与业务逻辑耦合严重、网络请求分散、资源和代码依赖复杂等问题，导致功能迭代效率下降，缺陷定位和维护成本较高。针对上述问题，我们结合项目特点采用了 C/S 架构、分层架构、MVC 架构、模块化架构和事件驱动架构等多种架构风格：客户端负责表现和交互，服务端负责核心数据和业务规则；客户端内部按表现层、业务逻辑层、数据层和基础设施层进行划分；UI 与数据通过观察者和事件机制解耦；公共能力通过程序集和接口进行模块化封装。重构后，系统模块边界更加清晰，代码耦合度降低，编译和迭代效率提升，后续活动、UI 和网络功能扩展更加稳定可控。
```

## 4. 建议修改后的提纲

```md
## 考场提纲 <span style="font-size: 0.75em; color: #666;">（约 450 字）</span>

说明：提纲用于考前构思和考场草稿，不计入论文正文。

一、项目背景

- 基于 Unity 的 3D 移动游戏项目，包含角色、战斗、UI、资源、网络、活动配置等模块。
- 客户端与服务端通过接口进行数据同步和业务交互。

二、遇到的问题

- 目录结构混乱，模块职责不清。
- UI、数据、业务逻辑耦合严重。
- 网络请求分散，公共能力重复实现。
- 资源、代码依赖复杂，编译和维护成本高。

三、架构目标

- 降低模块耦合，提高可维护性。
- 明确客户端与服务端职责。
- 建立分层结构和模块边界。
- 支持后续功能快速迭代。

四、关键措施

- 采用 C/S 架构，客户端负责表现和交互，服务端负责核心数据和规则。
- 采用分层架构，划分表现层、业务逻辑层、数据层、基础设施层。
- 采用 MVC 架构，实现 UI、数据、控制逻辑分离。
- 采用事件驱动架构，通过事件分发降低模块直接依赖。
- 使用 asmdef 和接口封装进行模块化管理。

五、实施效果

- 模块边界清晰，耦合度降低。
- UI、数据和逻辑分离后，功能修改影响范围变小。
- 网络请求和公共能力统一封装，重复代码减少。
- 编译速度和开发效率提升。

六、总结反思

- 架构风格选择要结合业务特点和团队能力。
- 客户端项目同样需要系统化架构设计。
- 架构重构要分阶段推进，避免一次性大改影响交付。
```

## 5. 正文建议结构

建议正文按以下结构展开：

```md
## 正文

### 一、项目背景

说明项目是什么、有哪些模块、为什么需要架构设计。

### 二、面临的主要问题

说明目录混乱、模块职责不清、UI 与业务耦合、网络请求分散、资源依赖复杂等问题。

### 三、架构风格的选择

说明为什么选择 C/S、分层、MVC、事件驱动、模块化架构。

### 四、架构风格的具体应用

详细写每种架构风格怎么落地。

### 五、实施效果

说明重构后可维护性、可扩展性、开发效率、编译效率等提升。

### 六、总结反思

说明架构风格不是越多越好，而是要结合项目特点进行组合和取舍。
```

## 6. 后续修改步骤

建议按以下顺序修改：

1. 删除当前正文中的负载均衡残留内容。
2. 替换项目背景表达，提升到“客户端及在线服务接入架构”层级。
3. 修改摘要，明确写出 C/S、分层、MVC、事件驱动、模块化架构。
4. 修改提纲，让每一项都围绕“架构风格”展开。
5. 再开始写正文，不要直接从现有负载均衡模板改。
