## 正文

### 一、项目背景 <span style="font-size: 0.75em; color: #666;">（367 字）</span>

我所在团队负责建设一套移动游戏运营知识与客服智能问答平台，为多款手机游戏提供统一知识查询能力。客户端由 Unity 和 Cocos 开发，服务端采用分布式架构，核心业务服务基于 Orleans Actor 模型实现，MySQL 存储账号、角色、订单、道具和活动记录等核心数据。平台涉及账号登录、商城支付、活动配置、排行榜、邮件公告、数据统计和运营后台等多个模块。

在实际运营中，知识资料分散在策划文档、接口文档、活动配置表、客服 FAQ、历史工单、支付接入说明和运维手册中。运营人员需要快速确认活动规则，客服人员需要解释充值、道具、封号和补偿问题，测试人员需要根据需求文档生成测试点。早期主要依赖人工查文档和询问开发人员，效率低且容易遗漏。

我们尝试引入大模型提升知识服务效率，但发现直接让大模型回答并不可靠。模型可能编造不存在的活动规则，也可能无法识别文档时效，还可能把无权限资料返回给普通运营人员。因此，我们决定建设一套基于 RAG 的智能问答系统，让大模型基于企业内部知识回答，并保证答案有来源、权限可控、过程可审计。

### 二、架构设计目标 <span style="font-size: 0.75em; color: #666;">（385 字）</span>

本系统的设计目标不是简单接入一个大模型接口，而是建设可在生产环境使用的企业知识服务架构。

第一，答案准确。系统必须优先基于企业内部文档、业务数据库和知识图谱回答，避免凭空生成。第二，来源可追溯。对于活动规则、支付流程、道具补偿等关键答案，必须返回引用来源，方便用户核对。第三，权限可控制。不同角色只能访问自己有权限的游戏项目、服务器、活动和运营资料。第四，知识可更新。新增活动文档、接口说明或客服 FAQ 后，应尽快进入可检索状态。第五，服务可监控。系统需要记录用户问题、检索片段、模型输出、响应时间和用户反馈，用于审计和持续优化。第六，成本和延迟可接受。热门问题不能每次都完整调用大模型和多路检索，需要通过缓存、重排和模型分级降低成本。

围绕这些目标，我们将系统划分为接入层、权限层、知识处理层、检索层、编排层、模型服务层、数据层和运维治理层。接入层面向运营后台、客服系统和企业 IM；权限层负责统一认证和数据权限过滤；知识处理层负责文档解析、清洗、切分、标注和向量化；检索层负责向量检索、关键词检索和知识图谱检索；编排层负责问题改写、结果重排和 Prompt 组装；模型服务层负责调用大模型和专有小模型；治理层负责监控、评测、审计和反馈闭环。

### 三、关键架构方案 <span style="font-size: 0.75em; color: #666;">（971 字）</span>

第一，建设统一知识处理流水线。系统从文档库、运营后台、客服系统和接口管理平台采集数据，对 Word、PDF、Markdown、表格和网页内容进行解析。解析后先进行清洗，去除页眉页脚、重复导航、无效图片说明和过期内容，再按照语义段落进行切分。每个知识片段都带上游戏项目、版本号、服务器范围、活动编号、文档来源、更新时间、密级和可访问角色等元数据。这样做的目的是让后续检索不仅能找到相似内容，还能判断内容是否有效、是否有权限访问。

第二，采用向量数据库和关键词检索结合的混合检索。用户提问通常不是直接使用文档原词，例如客服会问“玩家充值没到账怎么办”，文档中可能写的是“支付成功但发货失败处理流程”。因此我们将知识片段向量化后写入向量数据库，用于语义相似度检索。同时保留关键词索引，用于活动编号、道具 ID、错误码和接口名等精确检索。对于游戏业务，精确标识很多，如果只使用向量检索，可能漏掉关键错误码；如果只使用关键词检索，又无法理解自然语言问题。混合检索可以兼顾语义匹配和精确匹配。

第三，引入知识图谱表达业务关系。我们将游戏、活动、道具、订单、支付渠道、接口、错误码、处理流程等抽象为实体，并建立“活动包含奖励”“支付渠道触发回调”“错误码对应处理方法”“道具属于礼包”等关系。知识图谱主要用于关系查询和规则校验。例如用户询问某活动奖励发放失败时，系统可以沿着活动、奖励、道具、邮件和补偿流程查找相关知识，而不是只返回相似文本。知识图谱也用于校验模型输出，避免模型给出与业务关系冲突的答案。

第四，设计完整的 RAG 问答流程。用户提交问题后，系统先进行身份认证和权限识别，然后对问题进行意图识别、关键词提取和问题改写。系统在向量库、关键词索引和知识图谱中进行混合检索，再根据相关性、权限、文档时效和业务优先级进行重排。编排层将排名靠前的知识片段、引用来源、用户角色、回答约束和拒答规则拼接到 Prompt 中，调用大模型生成答案。答案返回时同时展示引用来源、更新时间和置信提示。如果检索结果不足，系统要求模型明确回答“资料不足，无法确认”，而不是自由编造。

第五，通过模型网关统一管理大模型调用。模型网关负责模型路由、限流、重试、超时控制、调用日志和成本统计。对于简单 FAQ 和高频问题，系统优先使用缓存或小模型；对于复杂问题，再调用能力更强的大模型。这样既能降低延迟和成本，也便于后续替换模型供应商或接入私有化模型。

第六，建立权限控制和安全审计机制。文档入库时按项目、角色、部门和密级打标签，检索前根据用户身份限定可检索范围，检索后再次过滤无权限片段，确保无权限内容不会进入 Prompt。账号、支付流水、设备号和玩家联系方式等敏感数据在入库和展示时进行脱敏。系统记录问题、检索内容、模型输出、用户反馈和调用链路，便于追踪错误答案和安全问题。

第七，建立评测和反馈闭环。我们整理了活动规则、支付异常、道具补发、接口错误和后台操作等典型问题作为评测集，定期评估答案准确率、引用准确率、拒答率和平均响应时间。用户可以对答案进行有用、无用和错误反馈，反馈结果进入人工复核流程。确认错误的问题会补充到知识库或规则库中，形成持续优化机制。

### 四、实施过程中的问题和处理 <span style="font-size: 0.75em; color: #666;">（508 字）</span>

第一个问题是文档质量不一致。部分策划文档缺少版本号，活动规则在聊天记录和配置表中不一致。我们要求知识入库前必须补充项目、版本、时间和责任人等元数据，并对过期文档设置失效标记。对于冲突内容，系统优先采用正式发布文档和线上配置。

第二个问题是检索结果相关但不一定可用。向量检索有时会返回语义相近但版本不匹配的活动说明。我们在重排阶段加入版本、服务器范围和更新时间权重，并在 Prompt 中要求模型只基于当前有效资料回答。

第三个问题是大模型幻觉。即使提供了上下文，模型仍可能补充没有依据的步骤。我们通过引用来源约束、缺少依据时拒答、知识图谱校验和离线评测降低幻觉风险。对支付、补偿和封号等高风险答案，系统只提供处理建议，最终操作仍要求人工确认。

第四个问题是权限泄露风险。客服人员、运营人员和开发人员能访问的资料不同。如果权限过滤只在答案返回前做，模型可能已经看到无权限内容。我们将权限控制前置到检索阶段，并在 Prompt 拼接前再次过滤，保证无权限片段不会进入模型上下文。

第五个问题是响应延迟和成本。完整 RAG 流程涉及多路检索和大模型调用，高峰时成本较高。我们对热门问答建立缓存，对简单问题采用小模型或模板回答，对复杂问题才调用大模型。同时对模型调用设置超时和降级策略，保证客服系统可用。

第六个问题是答案责任边界。AI 输出不能替代正式业务审批。我们在界面上明确展示来源、更新时间和置信提示，对补偿、封号、退款等高风险操作只生成建议，不直接执行。

### 五、实施效果 <span style="font-size: 0.75em; color: #666;">（335 字）</span>

系统上线后，运营和客服知识查询效率明显提升。常见活动规则、支付异常、道具补发和后台操作问题可以在几秒内返回答案，并附带引用来源。新人不再完全依赖老员工口头指导，客服处理重复问题的时间减少。

在一次大型活动中，部分玩家反馈奖励未到账。客服人员通过智能问答系统查询活动奖励发放流程，系统返回了活动配置、邮件发放机制、补偿处理步骤和相关接口说明，并标明文档来源。客服人员据此快速判断问题属于邮件队列延迟，而不是支付失败，避免了错误补偿。

同时，RAG 架构降低了大模型幻觉风险。对于知识库中没有明确依据的问题，系统会提示资料不足，而不是直接编造答案。权限过滤和审计日志也降低了数据泄露风险。新增活动文档经过解析、切分和向量化后，可以较快进入可检索状态。

当然，系统仍需要持续治理。知识库质量、文档更新及时性、评测集覆盖率和反馈处理效率会直接影响问答效果。我们将其纳入日常运维，通过监控、人工评审和知识库更新持续优化。

### 六、总结 <span style="font-size: 0.75em; color: #666;">（197 字）</span>

基于大模型的智能问答系统架构设计，关键不在于简单调用模型接口，而在于围绕企业知识建立可控、可追溯、可治理的工程体系。在本项目中，我们通过文档治理、向量数据库、知识图谱、混合检索、RAG 编排、模型网关、权限过滤、日志审计和反馈评测等措施，提升了移动游戏运营和客服知识服务效率。实践表明，大模型能够提高知识获取和问题处理效率，但必须重视幻觉控制、权限安全、来源追溯和持续评测。架构师需要在效果、成本、延迟、安全和可维护性之间做出合理权衡。
