论 Serverless 架构模式 <span style="font-size: 0.75em; color: #666;">（全文约 2960 字，提纲不计）</span>

## 考场提纲 <span style="font-size: 0.75em; color: #666;">（620 字）</span>

说明：提纲用于考前构思和考场草稿，不计入论文正文。

一、项目背景

- 项目：面向移动游戏的在线运营支撑平台。
- 模块：活动配置、礼包发放、支付回调、邮件通知、日志采集、运营报表、图片资源处理。
- 技术背景：Unity/Cocos 客户端、Orleans 服务端、MySQL、Python 脚本、云平台函数计算。
- 问题：活动流量波动大、后台任务多、部分服务长期空闲、运维成本高。

二、Serverless 核心组成

- FaaS：函数即服务，按事件触发执行业务逻辑。
- BaaS：后端即服务，如对象存储、消息队列、API 网关、云数据库、日志服务。
- 事件源：HTTP 请求、消息、定时任务、对象存储变更、数据库变更。
- 监控与权限：日志、指标、链路追踪、访问控制。

三、方案设计

- API 网关承接运营后台和外部回调请求。
- 云函数处理支付回调校验、礼包码校验、邮件异步发送、日志清洗。
- 消息队列削峰，异步处理非核心任务。
- 对象存储和云数据库承载资源、配置和任务状态。
- 监控告警保障函数错误率、执行耗时和积压量可见。

四、选型依据

- 业务具有明显峰谷。
- 部分任务短时、离散、事件触发明显。
- 不适合把所有任务都放进常驻服务。
- 需要降低闲置资源成本和运维负担。

五、难点与措施

- 冷启动：预热、拆小函数、控制依赖。
- 状态管理：状态外置到数据库和缓存。
- 调试困难：统一日志、请求 ID、链路追踪。
- 一致性：幂等、重试、补偿。

六、效果

- 后台任务运维成本下降。
- 活动峰值期间可自动弹性扩展。
- 非核心任务与核心游戏服务解耦。
- 资源按量使用，闲时成本降低。

## 摘要 <span style="font-size: 0.75em; color: #666;">（330 字）</span>

我参与设计并建设了一套面向移动游戏的在线运营支撑平台，服务于多款 Unity/Cocos 开发的手机游戏。平台主要包括活动配置、礼包发放、支付回调、邮件通知、日志采集、运营报表和资源处理等模块。随着游戏活动增多，系统出现了任务类型分散、流量峰谷明显、部分服务长期空闲、运维成本较高等问题。针对这些问题，我们在部分非核心但弹性需求明显的业务中引入 Serverless 架构模式，通过 API 网关、函数计算、消息队列、对象存储、云数据库和日志监控等组件，将支付回调校验、活动任务结算、邮件异步发送、日志清洗和定时统计等任务改造为事件触发的云函数。本文结合该项目，论述 Serverless 架构的核心组成、各组件作用、协同实现降本增效和弹性伸缩的过程，以及方案选型、落地难点和实施效果。实践表明，Serverless 适合短时、离散、事件驱动和峰谷明显的业务场景，能够降低闲置资源消耗和运维负担，但也需要重点处理冷启动、状态外置、可观测性和幂等补偿等问题。

## 正文

### 一、项目背景 <span style="font-size: 0.75em; color: #666;">（430 字）</span>

我所在团队负责建设一套面向移动游戏的在线运营支撑平台，为多款手机游戏提供统一的活动、支付、邮件、日志和运营分析能力。客户端主要由 Unity 和 Cocos 开发，核心游戏服务基于 Orleans 分布式框架实现，MySQL 用于存储账号、订单、道具、活动记录等核心数据，Python 脚本用于运营工具和数据处理任务。

在平台早期，很多后台任务都部署为常驻服务或定时脚本，例如支付回调校验、礼包码核销、活动奖励结算、邮件批量发送、日志清洗、运营报表生成等。这些任务有一个共同特点：平时请求量不高，但在活动开启、版本更新或支付高峰时会出现短时间突增。如果为这些任务长期部署足够的服务器，闲时资源浪费明显；如果按平均容量部署，峰值时又容易出现处理积压。此外，大量小型后台服务也增加了部署、监控和运维成本。

因此，我们没有将所有业务都迁移到 Serverless，而是选择事件触发明显、状态依赖较弱、对弹性要求高的非核心链路进行 Serverless 改造，使其与常驻核心游戏服务形成互补。

### 二、Serverless 架构的核心组成 <span style="font-size: 0.75em; color: #666;">（520 字）</span>

Serverless 并不是没有服务器，而是由云平台屏蔽服务器采购、部署、扩容和运行维护等底层细节，开发者主要关注业务函数和事件触发关系。在本项目中，Serverless 架构主要由 FaaS、BaaS、事件源、API 网关、监控告警和权限控制等部分组成。

FaaS 即函数即服务，是 Serverless 的核心执行单元。我们将支付回调校验、活动结算、邮件发送、日志清洗、报表生成等逻辑拆分为独立函数。函数由 HTTP 请求、消息队列、定时任务或对象存储事件触发，执行完成后自动释放运行资源。

BaaS 即后端即服务，提供函数运行所需的通用后端能力，包括对象存储、云数据库、消息队列、日志服务、配置服务和鉴权服务等。例如活动配置文件存放在对象存储中，函数读取配置后进行活动结算；异步邮件任务通过消息队列投递；任务状态和补偿记录写入数据库。

事件源负责触发函数执行。运营后台请求通过 API 网关触发函数，支付平台回调通过 HTTP 触发函数，活动结算通过定时任务触发函数，日志文件上传到对象存储后触发清洗函数。监控告警则用于观察函数执行次数、错误率、平均耗时、超时次数和队列积压量。权限控制用于限制函数访问数据库、对象存储和消息队列的范围，避免权限过大带来安全风险。

### 三、Serverless 方案设计与协同实现 <span style="font-size: 0.75em; color: #666;">（820 字）</span>

第一，在接入层使用 API 网关统一承接外部请求。运营后台、支付平台和内部工具不直接访问云函数，而是通过 API 网关进行鉴权、限流、参数校验和路由转发。对于支付回调这类重要入口，网关先进行基础校验，再将请求转发给支付回调函数。这样既降低了函数直接暴露的安全风险，也便于统一进行流量控制。

第二，将短时后台任务拆分为独立云函数。以活动奖励结算为例，活动服务在活动结束后产生结算任务，任务信息写入消息队列。结算函数被消息触发后，读取活动配置和玩家参与记录，计算奖励结果，并将邮件任务再次写入队列。邮件发送函数异步消费邮件任务，完成邮件创建和发送状态记录。通过这种方式，活动结算、奖励发放和邮件通知不再阻塞核心游戏服务。

第三，通过消息队列实现削峰和解耦。活动开始和结束时，玩家领取奖励、积分变动和排行榜更新会产生大量事件。如果直接同步处理所有后续逻辑，容易影响用户请求响应。我们将日志写入、运营统计、邮件通知等非核心任务投递到消息队列，由云函数按并发能力逐步消费。队列既起到缓冲作用，也使不同函数之间不发生强依赖。

第四，将状态外置到数据库、缓存和对象存储中。Serverless 函数通常是无状态的，不能依赖本地内存保存长期状态。因此，我们将订单状态、任务状态、幂等记录、补偿记录等保存到 MySQL 或云数据库中，将活动配置和日志文件放到对象存储中。函数每次执行都从外部存储读取必要状态，保证多实例并发执行时结果一致。

第五，建立统一监控和补偿机制。Serverless 降低了服务器运维成本，但并不意味着不需要运维。我们为每类函数设置执行耗时、错误率、重试次数和队列积压告警。对于支付回调、道具发放、奖励结算等关键任务，函数必须支持幂等处理，避免重试造成重复发放；当多次执行失败时，任务状态会进入待补偿表，由运营后台或运维工具触发人工补偿。

### 四、选型依据、难点和应对措施 <span style="font-size: 0.75em; color: #666;">（620 字）</span>

我们选择 Serverless 的主要依据是业务场景是否具有事件触发、短时执行、状态依赖弱和峰谷明显等特点。对于活动结算、日志清洗、定时报表、图片处理、邮件异步发送等任务，Serverless 能够较好地发挥按需启动和自动扩缩容的优势。而对于核心战斗服务、长连接网关、玩家状态管理等需要持续运行和低延迟响应的模块，我们仍然保留常驻服务。

落地过程中遇到的第一个难点是冷启动。部分函数首次执行时需要加载运行环境和依赖库，会带来额外延迟。对此，我们控制函数依赖规模，将大函数拆分为小函数，并对活动前可能高频调用的函数进行预热。

第二个难点是状态管理。函数实例可能随时销毁，不能依赖本地状态。我们将状态统一外置，并通过幂等键、唯一约束和任务状态机控制并发执行。

第三个难点是问题定位。传统服务可以登录机器查看日志，而 Serverless 更依赖平台日志和链路追踪。我们为每次请求生成统一请求 ID，并在 API 网关、函数、消息队列和数据库操作中传递该 ID，便于排查问题。

第四个难点是成本控制。Serverless 按量计费并不等于一定便宜，如果函数执行时间长、调用频率极高或日志量过大，成本也会快速上升。因此，我们对函数执行时间、并发上限和日志级别进行了控制。

### 五、实施效果 <span style="font-size: 0.75em; color: #666;">（310 字）</span>

通过引入 Serverless 架构，平台在运营后台和异步任务场景中取得了较明显效果。首先，活动结算、邮件发送、日志清洗等任务从常驻服务中剥离，核心游戏服务的压力降低。其次，活动峰值期间函数能够根据事件量自动扩展，任务积压情况比原来更容易控制。再次，很多低频任务不再需要单独维护服务器，闲时资源成本和运维工作量下降。

同时，Serverless 也带来了新的治理要求。我们必须更加重视函数拆分粒度、冷启动、权限边界、日志追踪、幂等和补偿机制。如果缺少这些设计，Serverless 可能会使问题更分散、更难排查。因此，Serverless 更适合作为整体架构中的弹性补充，而不是无条件替代所有服务。

### 六、总结 <span style="font-size: 0.75em; color: #666;">（220 字）</span>

Serverless 架构模式适合事件驱动、短时执行、峰谷明显和运维成本敏感的业务场景。在本项目中，我们将其用于移动游戏运营支撑平台中的活动结算、支付回调、邮件发送、日志清洗和定时报表等任务，通过 FaaS、BaaS、API 网关、消息队列和监控告警协同实现降本增效和弹性伸缩。实践表明，Serverless 能够降低闲置资源消耗和基础设施运维负担，但也需要处理冷启动、状态外置、可观测性和一致性补偿等问题。架构设计的关键不是追求某种模式先进，而是根据业务特点选择合适的运行形态。
