## 正文

### 一、项目背景 <span style="font-size: 0.75em; color: #666;">（276 字）</span>

我所在团队负责建设一套移动游戏高并发在线服务平台，为多款手机游戏提供统一的服务端能力。客户端由 Unity 和 Cocos 开发，服务端采用分布式架构，核心业务基于 Orleans Actor 模型实现，MySQL 存储账号、角色、订单、道具和活动记录等核心数据。平台主要模块包括登录、角色、匹配、活动、排行榜、商城支付、邮件和运营后台。

该平台具有明显的流量峰值特征。开服、版本更新、限时活动和排行榜结算时，大量玩家会同时登录、拉取配置、领取奖励和查询排行榜。普通无状态服务可以通过增加实例分担压力，但玩家、房间和匹配等有状态业务不能简单随机分发，否则会带来状态冲突和同步成本。因此，我们需要结合不同业务场景设计负载均衡方案。

### 二、负载均衡类型和常用方法 <span style="font-size: 0.75em; color: #666;">（381 字）</span>

负载均衡的目标是将请求、任务或数据访问合理分配到多个处理节点，避免单点过载，提高系统吞吐量、可用性和扩展能力。

静态负载均衡是按照预先设定的规则进行分配，不依赖实时负载状态。常见方法包括轮询、加权轮询、随机、加权随机和哈希。它实现简单、开销小，适合实例性能接近、请求处理成本差异不大的无状态服务。

动态负载均衡是根据节点实时状态调整分配策略。它会参考健康检查、响应时间、连接数、CPU 使用率、错误率等指标，将更多流量分配给健康且负载较低的节点，将异常节点摘除或降低权重。它适合服务实例状态波动明显或请求耗时差异较大的场景。

基于场景的负载均衡是根据业务特点选择不同策略。登录服务关注高并发接入，排行榜关注热点查询，匹配服务关注队列状态，支付服务关注可靠性和幂等性，玩家 Actor 服务关注状态一致性。不同场景不能简单套用同一个算法。

常用实现方法包括网关层分发、服务注册与发现、健康检查、故障摘除、一致性哈希、业务分片、读写分离、缓存和动态扩缩容等。

### 三、负载均衡方案设计 <span style="font-size: 0.75em; color: #666;">（556 字）</span>

第一，在接入层使用网关进行流量分发。移动客户端请求首先进入 API 网关，网关负责鉴权、限流、协议适配和请求转发。对于登录、公告、配置拉取等无状态请求，网关采用轮询或加权轮询分发到多个服务实例。对于健康检查失败、响应时间过长或错误率升高的实例，网关会临时摘除，避免异常实例继续接收流量。

第二，在服务层区分无状态和有状态业务。账号、公告、配置等无状态服务可以通过增加实例横向扩展。玩家状态、房间状态和匹配队列等有状态业务则结合 Orleans Actor 模型，通过玩家编号、房间编号或匹配分区进行路由，使同一对象的请求由对应 Actor 处理，避免并发修改状态。

第三，对活动和排行榜场景进行业务分区。活动开始时，玩家请求集中，如果所有请求访问同一个活动服务或数据库表，容易形成热点。我们按服务器分区、活动编号和用户范围进行拆分，使请求分散到不同处理单元。排行榜展示数据放入缓存，并按榜单类型和服务器分区管理，减少数据库直接访问。

第四，在数据层采用读写分离和缓存。MySQL 主库负责订单、道具、支付等强一致写入，只读实例和缓存承接角色查询、活动配置和排行榜展示等读请求。对于一致性要求高的数据仍访问主库，对于展示类数据可以接受短暂延迟。

第五，结合监控实现动态调整。我们监控服务实例 QPS、响应时间、错误率、CPU、内存、队列积压和数据库连接数。当某个实例负载过高或出现异常时，降低其权重或摘除；活动开始前根据预估流量提前扩容，活动结束后再缩容。

### 四、选型依据和实施难点 <span style="font-size: 0.75em; color: #666;">（385 字）</span>

负载均衡方案的选择依据主要包括服务是否有状态、请求处理成本是否均衡、流量是否存在峰值、数据是否存在热点以及业务是否要求强一致。无状态服务适合轮询和动态权重，有状态服务适合按业务 ID 路由，排行榜和活动适合缓存和分区，支付服务则优先保证可靠性和幂等性。

落地过程中，第一个难点是有状态服务的热点问题。按玩家或房间路由可以保证状态一致，但部分大公会、大房间或热门活动可能集中到少数实例。我们通过业务分区、热点监控和局部迁移缓解该问题。

第二个难点是健康检查准确性。如果健康检查过于简单，可能无法发现慢响应；如果过于敏感，又可能误摘除正常实例。我们结合接口探测、错误率和响应时间综合判断。

第三个难点是缓存一致性。排行榜和活动配置放入缓存后，需要处理更新延迟和失效策略。我们通过版本号、过期时间和主动刷新控制风险。

第四个难点是扩容效果验证。增加实例并不一定自动提升容量，如果瓶颈在数据库或缓存，服务扩容效果有限。因此需要结合压测和监控定位真实瓶颈。

### 五、实施效果 <span style="font-size: 0.75em; color: #666;">（222 字）</span>

通过负载均衡设计，平台在活动高峰期的稳定性明显提升。无状态服务可以通过增加实例分担请求，有状态玩家和房间请求通过 Actor 路由保持一致性，活动和排行榜热点通过缓存和分区得到缓解，异常实例能够通过健康检查及时摘除。数据库读压力也通过只读实例和缓存得到分担。

同时，我们认识到负载均衡不是简单地把请求平均分配。对于移动游戏服务平台，不同业务模块的状态特征、数据一致性要求和访问热点差异很大，必须采用组合策略。负载均衡还需要与监控、限流、缓存、扩容和降级共同设计，才能真正提升系统可用性。

### 六、总结 <span style="font-size: 0.75em; color: #666;">（187 字）</span>

负载均衡技术是高并发系统架构设计中的重要手段。在本项目中，我们围绕移动游戏服务平台的业务特点，将静态负载均衡、动态负载均衡和基于场景的负载均衡结合使用，在接入层、服务层、数据层和业务场景中分别采用不同策略。实践表明，合理的负载均衡能够提升系统吞吐量、扩展能力和故障隔离能力，但架构师必须充分考虑有状态业务、热点数据、健康检查、缓存一致性和数据库瓶颈。负载均衡的本质是在业务正确性和资源利用效率之间取得平衡。
